# โครงรายงาน Multi-agent LLM คุ้มกว่าเอเจนต์เดี่ยวไหม? กรณีศึกษาการตรวจจับประชดภาษาไทย

> ตัวเลขทุกตัวในโครงนี้มาจากไฟล์ผลจริงในโฟลเดอร์นี้ ไม่มีตัวเลขสมมติ
> แหล่งอ้างอิง: `baseline_preds_gpt.csv`, `multiagent_preds_gpt_conservative.csv`,
> `multiagent_preds_gpt_v1aggressive.csv`, `compare_systems.py`, `PROVENANCE.md`

## 1. บทนำ / คำถามวิจัย

**คำถาม:** ระบบ multi-agent (แบ่งงานให้ LLM หลายด่าน) คุ้มกว่าการเรียก LLM ครั้งเดียวไหม
**คุ้ม = วัด 4 มิติพร้อมกัน:** คุณภาพ (F1) · ค่าใช้จ่าย · เวลา · จำนวน LLM calls

**ทำไมต้องเป็นงานนี้:** ประชดภาษาไทยเป็นงานที่ "ความหมายผิวเผินตรงข้ามกับเจตนา"
ซึ่งเป็นโจทย์ที่ควรได้ประโยชน์จากการให้ LLM คิดเป็นหลายด่าน ถ้า multi-agent
ไม่ช่วยแม้แต่ในงานแบบนี้ ก็เป็นหลักฐานที่หนักแน่นว่ามันไม่ใช่ยาครอบจักรวาล

**นิยามที่ใช้ (จาก `labeling_rubric.md`):**
ประชด = ต้องมี **"การเสแสร้ง"** (แกล้งชม/แกล้งขอบคุณเพื่อเหน็บ)
- บ่นตรงๆ ≠ ประชด
- รีวิวสมดุล (ชมจริง + ติจริง) = ไม่ประชด (0)
นิยามนี้สำคัญมาก เพราะมันคือต้นตอของ finding หลักทั้งหมด (ดูข้อ 5)

## 2. ชุดข้อมูล (gold set)

| | |
|---|---|
| แหล่งข้อมูล | Wongnai (รีวิวร้านอาหาร, ยาว) + Wisesight (ทวีต, สั้น) |
| ขนาด | **127 ข้อ** |
| ประชด (1) | **30 ข้อ** (23.6%) |
| ไม่ประชด (0) | **97 ข้อ** (76.4%) |

**วิธีสร้าง (ต้องเล่าตรงๆ ดู `PROVENANCE.md`):**
ข้อมูลดิบ → คัดด้วยคีย์เวิร์ด → **GPT-4o ช่วยขุดเพิ่ม (คัด 800 → 495 ข้อ)**
→ Claude จัดอันดับ (130 → 37) → **คนตัดสินขั้นสุดท้ายตาม rubric แบบ blind**
(ไม่เห็น confidence ของ LLM ตอนติดป้าย เพื่อไม่ให้ LLM ชี้นำคน)

**ข้อจำกัดที่ต้องประกาศเอง self-selection bias:**
ฝั่ง "ประชด" ไม่ได้มาจากการสุ่ม แต่ผ่านการคัดกรองของ GPT-4o มาก่อน
→ **recall ของ GPT-4o ในรายงานนี้สูงเกินจริง** (เพราะเราไปตักข้อที่มันหาเจออยู่แล้ว)
→ แต่ bias นี้ **กระทบทุกระบบเท่ากัน** (ทั้ง 3 ระบบใช้ GPT-4o ทั้งหมด)
   ดังนั้น**การเทียบระบบต่อระบบยังยุติธรรม** สิ่งที่เชื่อไม่ได้คือ "ตัวเลขดิบ" ไม่ใช่ "ผลต่าง"

**หลักฐานเชิงตัวเลขของ bias นี้ (จาก `analyze_baseline.py`):**
มันโผล่ที่ **false positive rate ไม่ใช่ recall**
- ข้อที่มาจากคีย์เวิร์ด: FPR = 0.205
- ข้อที่มาจากการขุดโดย LLM: FPR = **1.000 (9/9)**
- recall = 1.000 เท่ากันทั้งสองกลุ่ม
→ ตีความ: กองที่ LLM ขุดมาคือกอง "ข้อที่ LLM คิดว่าประชด" ซึ่งคนบอกว่าไม่ใช่ มันเลยพลาดหมด

## 3. วิธีการ 3 ระบบ วัดบน harness เดียวกัน

ทุกระบบใช้ **GPT-4o** ตัวเดียวกัน, gold ชุดเดียวกัน, และ**โค้ดวัดผลตัวเดียวกัน**
(`multiagent.py` `import metrics` มาจาก `baseline.py` โดยตรง พิสูจน์ได้ว่าไม่มีการวัดคนละมาตรฐาน)

**① เอเจนต์เดี่ยว (baseline)**, `baseline.py`
เรียก LLM ครั้งเดียว → ตอบ JSON `{"label": "1"/"0"}`

**② Multi-agent: detector → verifier**, `multiagent.py`
- **detector** = prompt เดียวกับ baseline เป๊ะ (ตัวแปรควบคุม)
- **verifier** = ด่าน**กรองทิ้ง** false positive ตาม rubric **ไม่ใช่ด่านหาประชดเพิ่ม**
- verifier ทำงาน**เฉพาะข้อที่ detector ตอบ 1** เท่านั้น
  เหตุผล: verifier พลิกได้ทางเดียว (1→0) ยิงกับข้อที่ตอบ 0 อยู่แล้วก็เปลืองโทเคนเปล่า
  → นี่คือเหตุผลที่ calls = 183 ไม่ใช่ 254 (ประหยัดไป 28%)

**③ WangchanBERTa (ยังไม่ทำ งานต่อ)**
โมเดลเล็กเทรนเอง ตอบคำถาม "ต้องใช้ LLM จริงไหม"

**การจัดการ error (สำคัญต่อความน่าเชื่อถือ):**
ถ้า LLM ตอบเพี้ยน → บันทึกเป็น `"err"` **ไม่เดาเป็น `"0"`**
เพราะ gold เป็น 0 ถึง 76% การเดา 0 จะถูกโดยบังเอิญบ่อย → ตัวเลขสวยเกินจริง
(ผลจริง: **0 error ทั้ง 127 ข้อ ในทุกระบบ** ไม่มีข้อไหนถูกตัดทิ้ง)

## 4. ผลการทดลอง

### 4.1 ตารางหลัก 4 มิติครบ

| ระบบ | F1 | precision | recall | LLM calls | ค่าใช้จ่าย | latency p50 |
|---|---|---|---|---|---|---|
| **baseline** (เดี่ยว) | 0.690 | 0.526 | **1.000** | 127 | **$0.094** | **751 ms** |
| multi-agent **v1** (verifier ดุ) | 0.714 | **0.769** | 0.667 | 180 | $0.157 | 721 ms |
| multi-agent **v2** (verifier ระวัง) | **0.744** | 0.604 | 0.967 | 183 | $0.169 | 967 ms |

**ราคาที่ multi-agent v2 ต้องจ่ายเทียบ baseline:**
- ค่าใช้จ่าย **1.80×** (โทเคน 34,535 → 62,859 in)
- LLM calls **1.44×**
- latency **1.29× ที่ p50** (751 → 967 ms) / รวมทั้งชุด 105 วิ → 131 วิ

### 4.2 confusion matrix ของ baseline (ที่มาของทุกอย่าง)

|  | ทายว่าประชด | ทายว่าไม่ประชด |
|---|---|---|
| **ประชดจริง** | TP = **30** | FN = **0** |
| **ไม่ประชดจริง** | FP = **27** | TN = 70 |

→ **จับครบทุกข้อ (FN = 0) แต่เหวี่ยงแหเกิน 27 ข้อ**

### 4.3 การทดสอบนัยสำคัญ (paired bootstrap, 5000 รอบ)

**ทำไมต้อง paired:** ทั้งสองระบบวัดบน gold **ชุดเดียวกันข้อต่อข้อ**
การเทียบแบบ "F1 ต้องเกินขอบบน CI ของ baseline" คือการทำเหมือนวัดคนละชุด → **เข้มเกินไปผิดๆ**
ที่ถูกคือ resample ข้อเดียวกันทั้งคู่ แล้วดู**ผลต่าง** F1 ต่อรอบ

**ผล (v2 vs baseline):**
- ผลต่าง F1 = **+0.054**
- 95% CI ของผลต่าง = **[−0.002, +0.113]**
- โอกาสที่ v2 **ไม่**ดีกว่า = **3%**
- **McNemar: v2 ถูก/baseline ผิด = 8 ข้อ | baseline ถูก/v2 ผิด = 1 ข้อ**

→ **"ดีกว่าจริงด้วยความมั่นใจ ~97% แต่ยังไม่ผ่านเกณฑ์ 95% แบบสองด้านเป๊ะ"**
   ขอบล่างของ CI คร่อม 0 อยู่แค่ −0.002 เฉียดมาก

## 5. Findings (หัวใจของรายงาน)

### Finding 1 สมมติฐานตั้งต้นผิด: ปัญหาไม่ใช่ recall แต่เป็น precision

เราตั้งต้นคิดว่า LLM จะ "หาประชดไม่เจอ" (ประชดแนบเนียน)
**ผลจริงตรงข้าม: recall = 1.000 เต็ม จับครบทุกข้อ**
คอขวดคือ **precision 0.526** เกินครึ่งของที่มันบอกว่าประชด คือไม่ประชด

**FP 27 ข้อส่วนใหญ่คือ "รีวิวสมดุล"** (ชมจริงบางจุด ติจริงบางจุด)
ซึ่ง rubric ระบุชัดว่า = 0 เพราะ**ไม่มีการเสแสร้ง**
→ LLM สับสนระหว่าง **"มีทั้งบวกและลบในข้อความเดียว"** กับ **"แกล้งชมเพื่อเหน็บ"**
   นี่คือความเข้าใจผิดเชิงความหมายที่ชัดเจนและอธิบายได้ ไม่ใช่แค่ noise

### Finding 2 **การออกแบบ verifier สำคัญกว่าการ "มี" verifier** ⭐

นี่คือ finding ที่มีค่าที่สุด และเป็นสิ่งที่ทดลองแบบ ablation ได้จริง
**verifier ตัวเดียวกัน สถาปัตยกรรมเดียวกัน ต่างกันแค่กฎตอนก้ำกึ่ง:**

| | v1 "ไม่ชัด → พลิกทิ้ง" | v2 "ไม่ชัด → เก็บไว้" |
|---|---|---|
| พลิกทั้งหมด | 27 ข้อ | 8 ข้อ |
| ฆ่า FP ถูก | 17 | 7 |
| ฆ่า TP ผิด (เสียหาย) | **10** | **1** |
| recall | 0.667 | 0.967 |
| F1 | 0.714 | **0.744** |

**v1 แลก recall กับ precision เกือบ 1:1 → F1 แทบไม่ขยับ** (แค่ +0.024)
**v2 พลิกน้อยแต่แม่น → F1 +0.054**

**บทเรียน:** พูดว่า "ใส่ verifier แล้วดีขึ้น" ยังไม่พอ verifier ที่ออกแบบผิด
**เผา TP ทิ้งเร็วกว่าที่มันฆ่า FP** ทั้งที่จ่ายค่าโทเคนเท่ากัน
เหตุผลเชิงหลักการ: **ประชดที่แนบเนียนอ่านได้สองแง่โดยธรรมชาติ**
ดังนั้น "ถ้าไม่ชัดให้ทิ้ง" จึงเป็นกฎที่ขัดกับธรรมชาติของงานนี้โดยตรง
→ **prior ของ verifier ต้องสอดคล้องกับธรรมชาติของงาน ไม่ใช่ตั้งตามสัญชาตญาณ**

### Finding 3, multi-agent ช่วยได้จริง แต่ marginal และไม่ฟรี

F1 +0.054 (~97% confidence) แลกกับ **cost 1.80× · calls 1.44× · latency 1.29×**
"คุ้มไหม" ไม่มีคำตอบสากล ขึ้นกับว่างานนั้น FP แพงแค่ไหน
- ถ้า FP แพง (เช่น ระบบเตือน moderator): **คุ้ม**
- ถ้าแค่จัดหมวดหมู่ทั่วไป: **ไม่คุ้ม จ่ายเกือบเท่าตัวเพื่อ F1 5 จุด**

### Finding 4 **อย่ารายงาน accuracy** (ข้อควรระวังเชิงระเบียบวิธี)

baseline accuracy = **0.787**
เดาว่า "ไม่ประชด" ทุกข้อ = **0.764**
→ ต่างกัน 2 จุด ทั้งที่ระบบหนึ่งใช้ LLM อีกระบบไม่ทำอะไรเลย
**accuracy ซ่อนความล้มเหลวทั้งหมดไว้** เพราะข้อมูลเอียง 76/24

## 6. ข้อจำกัด (เขียนเองก่อนโดนถาม)

1. **Self-selection bias** gold ฝั่งประชดถูก GPT-4o คัดมาก่อน → ตัวเลข recall สูงเกินจริง
   (บรรเทา: bias กระทบทุกระบบเท่ากัน → การเทียบยังใช้ได้ / มีหลักฐานเชิงตัวเลขในข้อ 2)
2. **n ประชด = 30 น้อยเกินจะฟันธง** CI จึงกว้าง และ v2 พลาดเกณฑ์ 95% ไปแค่ 0.002
   **นี่คือเพดานทางสถิติ ไม่ใช่เพดานทางเทคนิค** จูน prompt เก่งแค่ไหนก็ไม่ทำให้ CI แคบลง
3. **ผู้ติดป้ายคนเดียว** ไม่มี inter-annotator agreement (κ) → วัด "ความเห็นตรงกันของคน" ไม่ได้
4. **โมเดลเดียว (GPT-4o)** ยังไม่รู้ว่า finding นี้ generalize ข้ามโมเดลไหม
5. **WangchanBERTa ยังไม่ได้ทำ** → ยังตอบไม่ได้ว่า "ต้องใช้ LLM จริงไหม"

## 7. งานต่อ (เรียงตามผลตอบแทน)

1. **ขยาย gold ฝั่งประชดเป็น 50–60 ข้อ** ← **คุ้มที่สุด**
   เป็นทางเดียวที่ทำให้ CI แคบพอจะฟันธงได้ (`harvest_to_review.csv` ยังเหลือ ~470 ข้อยังไม่ตรวจ)
   *และควรสุ่มตรวจข้อที่ LLM บอกว่า "ไม่ประชด" ด้วย เพื่อลด self-selection bias*
2. **WangchanBERTa** ตอบว่าโมเดลเล็กพอไหม
3. **แก้ precision ที่ต้นทาง** ใส่ตัวอย่าง "รีวิวสมดุล = 0" ลงใน prompt ของ detector
   (ถูกกว่าการเพิ่ม verifier ทั้งด่าน เพราะไม่เพิ่ม call เลย)
4. ผู้ติดป้ายคนที่สอง → รายงาน κ

## 8. บทสรุป

ในงานตรวจจับประชดภาษาไทย **multi-agent ให้ F1 ดีกว่าเอเจนต์เดี่ยวจริง (+0.054, ~97% confidence)
แต่ต้องจ่ายค่าใช้จ่าย 1.80 เท่าและช้าลง 1.29 เท่า** เป็นการแลกที่คุ้มเฉพาะเมื่อ false positive มีราคาแพง

และบทเรียนที่สำคัญกว่าตัวเลข: **การมี verifier ไม่ได้การันตีว่าจะดีขึ้น**
verifier ตัวเดียวกันที่ตั้งกฎตอนก้ำกึ่งต่างกัน ให้ recall ต่างกันถึง 0.667 vs 0.967
**สถาปัตยกรรม multi-agent ไม่ใช่ตัวแปรที่สำคัญ การออกแบบ prompt ของแต่ละด่านต่างหากที่สำคัญ**
