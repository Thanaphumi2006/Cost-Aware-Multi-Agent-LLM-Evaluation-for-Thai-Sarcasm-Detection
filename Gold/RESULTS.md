# ผลการทดลอง: เอเจนต์เดี่ยว vs multi-agent (ประชดภาษาไทย)

วัดบน gold 127 ข้อ (ประชด 30 / ไม่ประชด 97) ทุกระบบใช้ GPT-4o + harness เดียวกัน
ที่มาของ gold และ self-selection bias: ดู `PROVENANCE.md`

## สรุปผล 3 ระบบ

| ระบบ | F1 | precision | recall | LLM calls | ค่าใช้จ่าย | latency p50 |
|---|---|---|---|---|---|---|
| **baseline** (เดี่ยว) | 0.690 | 0.526 | **1.000** | 127 | $0.094 | 751 ms |
| **baseline + threshold** (เดี่ยว, ปรับ threshold จาก logprob) ⭐ | 0.725 | 0.641 | 0.833 | 127 | **$0.094** | 751 ms |
| multi-agent **v1** (verifier ตื้น) | 0.714 | **0.769** | 0.667 | 180 | $0.157 | 721 ms |
| multi-agent **v2** (verifier อนุรักษ์ recall) | **0.744** | 0.604 | 0.967 | 183 | $0.169 | 967 ms |
| **debate** (อัยการ+ทนาย+ผู้พิพากษา) | 0.694 | 0.595 | 0.833 | 381 | $0.695 | 4557 ms |
| **cascade** (WCB คัดกรอง → GPT verifier) ✗ | 0.628 ± 0.006 | 0.500 | 0.844 | 119 | $0.124 |, |
| **WangchanBERTa** (5-fold CV × 3 seeds) | 0.620 ± 0.005 | 0.553 | 0.700 | **0** | **$0.00** | **26 ms** |

WangchanBERTa วัดด้วย 5-fold stratified CV (ทุกข้อถูกทำนายโดยโมเดลที่ไม่เคยเห็นข้อนั้น)
→ ได้ pred ครบ 127 ข้อ เทียบ paired กับ LLM ได้ตรงๆ · เทรนบน CPU 44 นาที · รันออฟไลน์ได้

## finding หลัก (ตอบธีสิส "multi-agent คุ้มไหม")

**1. baseline มี recall เต็ม (1.000) แต่ precision ต่ำ (0.526)**
GPT-4o จับประชดครบทุกข้อ แต่ตีความ "รีวิวสมดุล (ชมจริงติจริง)" เป็นประชดผิดๆ 27 ข้อ
-> ปัญหาไม่ใช่ "หาประชดไม่เจอ" แต่เป็น "เหวี่ยงแหกว้างเกิน"

**2. verifier เป็นด่านกรอง FP -- ปรับ prompt แล้วผลต่างกันมาก**
- v1 (ไม่ชัดก็พลิกทิ้ง): พลิก 27 ข้อ -- ถูก 17 ผิด 10 -> recall ร่วงเหลือ 0.667, F1 แทบไม่ขึ้น
- v2 (พลิกเฉพาะที่ชัดว่าไม่ใช่ประชด): พลิก 8 ข้อ -- เสีย recall แค่ 1 ข้อ -> F1 ดีสุด 0.744

**3. v2 ดีกว่า baseline แบบ "เกือบมีนัยสำคัญ" แต่แลกด้วยค่าใช้จ่ายเกือบเท่าตัว**
paired bootstrap (resample gold ชุดเดียวกัน 5000 รอบ):
- ผลต่าง F1 = +0.054, 95% CI [-0.002, +0.113]
- CI คร่อม 0 อยู่นิดเดียว (ขอบล่าง -0.002) -> โอกาสที่ v2 ไม่ดีกว่า = แค่ 3%
- McNemar: v2 ถูก-baseline ผิด 8 ข้อ / baseline ถูก-v2 ผิด 1 ข้อ (เอียงไปทาง v2 ชัด)
=> สรุปได้ว่า "มีแนวโน้มดีกว่าจริง (~97% confidence)" แต่ยังไม่ผ่านเกณฑ์ 95% แบบเป๊ะ
   ที่ n=30 ประชด กำลังสถิติไม่พอจะฟันธง

**4. agent มากขึ้น != ดีขึ้น -- debate แพ้ pipeline ที่เรียบง่ายกว่า ทั้งที่แพงกว่า 4 เท่า**
สถาปัตยกรรม debate (อัยการ + ทนาย -> ผู้พิพากษา, 3 calls/ข้อ, ตัดสินใหม่ได้ทั้งสองทาง):
- F1 **0.694** -- แทบไม่ต่างจาก baseline เอเจนต์เดี่ยว (+0.005, โอกาสไม่ดีกว่า = **46%** = โยนหัวก้อย)
- แพ้ pipeline v2 (0.744): ผลต่าง **-0.049**, โอกาสที่ debate ไม่ดีกว่า v2 = **83%**
- **แต่จ่ายแพงกว่า v2 ถึง 4.1× ($0.695 vs $0.169) และช้ากว่า 4.7× (4557 vs 967 ms)**

สาเหตุที่ debate แพ้ -- **มันคืนอิสระให้ตัดสินใหม่ทั้งสองทาง แล้วเสีย recall ฟรีๆ**:
recall ร่วงจาก 1.000 (baseline) เหลือ **0.833** -- ทนายแก้ต่างสำเร็จให้ประชดจริง 5 ข้อ (FN=5)
ขณะที่ verifier ของ v2 **โครงสร้างบังคับ** ให้พลิกได้ทางเดียว (1->0) และตั้งกฎ "ไม่ชัดให้คงไว้"
=> v2 จึงรักษา recall 1.000 ที่ detector ได้มาฟรีไว้เกือบครบ (0.967) แล้วค่อยๆ ซื้อ precision

**บทเรียนเชิงสถาปัตยกรรม (สำคัญกว่าตัวเลข):**
**"ข้อจำกัดของ agent" มีค่ามากกว่า "จำนวน agent"**
verifier ที่ทำได้อย่างเดียวคือปัดตก (จำกัดมาก) เอาชนะคณะ 3 agent ที่ตัดสินอิสระได้ทุกทาง
เพราะมันหวงของดีที่มีอยู่แล้ว (recall เต็ม) แทนที่จะเอาไปเสี่ยงใหม่ทุกข้อ

**5. โมเดลเล็กฟรี "ยังตัดทิ้งไม่ได้" -- และนี่คือผลที่ไม่คาดคิดที่สุด**
WangchanBERTa (F1 0.620) ดูแพ้ baseline (0.690) ที่ผิวเผิน แต่พอทดสอบแบบ paired:
- ผลต่าง F1 = -0.072, 95% CI **[-0.206, +0.053]** -> **คร่อม 0**
- McNemar: WCB ถูก-baseline ผิด **15** ข้อ | baseline ถูก-WCB ผิด **14** ข้อ -> **แทบเสมอ**
=> ที่ n=30 **ยังฟันธงไม่ได้ว่า GPT-4o ชนะโมเดลฟรีที่เทรนด้วยข้อมูลแค่ 127 ข้อจริง**

แต่ **อย่าตีความเกินจริง**: point estimate ยังบอกว่า WCB แย่กว่า (โอกาสที่มันแย่กว่า = 87%)
"สรุปไม่ได้ว่าแย่กว่า" != "ดีเท่ากัน" -- มันแค่แปลว่า gold เล็กเกินกว่าจะตัดสิน
และ **error profile ต่างกันชัด**: WCB ปล่อยประชดหลุด 9 ข้อ (recall 0.700) ขณะที่ LLM ไม่หลุดเลย
-> ถ้างานคือ "คัดกรองห้ามพลาด" LLM ยังชนะขาด แม้ F1 จะไม่ต่างกันอย่างมีนัยสำคัญ

ข้อดีที่วัดได้ของ WCB: **$0.00 · 0 API calls · 26 ms (เร็วกว่า LLM 29 เท่า) · รันออฟไลน์ได้**

**6. เอา WangchanBERTa (ฟรี) มาเป็นด่านคัดกรองแทน GPT "ไม่ได้ผล" และเหตุผลที่ล้มเหลวคือ finding เอง**

ไอเดีย (ตรงตามบทเรียนข้อ 4 เป๊ะ): สถาปัตยกรรมที่ชนะคือ *ด่านคัดกรอง recall สูง → verifier ที่ปัดตกได้อย่างเดียว*
แต่ v2 ใช้ **GPT เป็นด่านคัดกรอง = จ่ายเงินทุกข้อ (127 calls)** ซึ่งเป็น "พื้นราคา" ที่ตัดไม่ได้เลย
ถ้าเปลี่ยนด่านแรกเป็น WangchanBERTa (ฟรี · ออฟไลน์ · 26 ms) พื้นราคานั้นควรหายไปทั้งชั้น

**F1 0.620 ของ WCB ไม่ใช่เหตุผลที่มันเป็นด่านคัดกรองไม่ได้** ด่านคัดกรองไม่ต้องแม่น ต้องแค่ "ไม่ปล่อยประชดหลุด"
(recall สูง) ส่วนที่เหวี่ยงเกิน verifier ตัดทิ้งให้ทีหลัง → จึงต้องวัดที่ **recall ตอนลด threshold** ไม่ใช่ F1 ที่ 0.5

วิธีวัด: ผลิต out-of-fold probability (`wcb_oof_probs.py`, 5-fold × 3 seeds โปรโตคอลเดียวกับข้อ 5)
แล้วเลือก threshold แบบ **leave-fold-out** (ข้อใน fold k ใช้ threshold ที่เลือกจากอีก 4 folds เท่านั้น)
→ ไม่มีข้อไหนมีส่วนกำหนด threshold ที่ตัดสินตัวมันเอง (กัน threshold leak)

| target recall | flag (จาก 127) | recall จริง | precision ด่านแรก | GPT calls | ค่าใช้จ่าย | vs v2 |
|---|---|---|---|---|---|---|
| 0.85 | 81 | 0.822 | 0.31 | 81 | $0.085 | 0.50× |
| 0.90 | 92 | 0.889 | 0.29 | 92 | $0.096 | 0.57× |
| 0.95 | 105 | 0.933 | 0.27 | 105 | $0.110 | 0.65× |
| 1.00 | **119** | 0.967 | **0.24** | 119 | $0.124 | 0.74× |

**ตัวเลขที่ฆ่าไอเดียนี้: เพื่อจับประชดให้ได้ 29/30 ข้อ ด่านคัดกรองต้อง flag 119 จาก 127 ข้อ = 94% ของทั้งชุด**

- ส่งทุกข้อให้ verifier ตรงๆ โดยไม่ต้องคัดกรองเลย = 127 calls, recall 1.000
- ให้ WCB คัดกรองก่อน = 119 calls, recall 0.967
- → **ด่านคัดกรองฟรีนี้ประหยัดได้ 8 calls แล้วทำประชดจริงหลุดไป 1 ข้อเป็นค่าตอบแทน**
  มันไม่ได้ "กรอง" อะไรเลย มันแค่เพิ่มขั้นตอน

เทียบกับ GPT ที่ทำงานเดียวกัน: recall 1.000 โดย flag แค่ **57 ข้อ** (precision 0.526)
WCB ต้อง flag มากกว่า **2 เท่า** เพื่อให้ได้ recall **ต่ำกว่า** ที่ precision ครึ่งเดียว (0.24)
และ verifier ของ v2 ถูกออกแบบให้อนุรักษ์นิยม ("ไม่ชัด → คงไว้") ใน v2 มันปัดตกแค่ 8 จาก 56 ข้อที่เห็น (14%)
→ โยน FP ให้มัน 89 ข้อแทนที่จะเป็น 27 ข้อ **มันกู้ precision ไม่ไหวแน่นอน**

**บทเรียน: "ฟรี" ไม่ได้แปลว่า "เอามาวางหน้า pipeline แล้วประหยัด"**
ด่านคัดกรองที่ใช้ได้ต้อง **จัดอันดับ (rank) เป็น** ต้องแยกได้ว่าข้อไหน "ชัดว่าไม่ประชด" เพื่อจะตัดทิ้งฟรี
WCB ที่เทรนด้วย 127 ข้อ **ทำไม่ได้**: คะแนนของมันแทบไม่มีข้อมูลตรงหางที่ต้องใช้ตัดสิน
→ ยืนยันข้อ 5 อีกทาง: โมเดลเล็กฟรี "ยังตัดทิ้งไม่ได้" ไม่ใช่แค่ในฐานะระบบเดี่ยว แต่ในฐานะ **ชิ้นส่วน** ของระบบด้วย

**ยืนยันด้วยการรันจริง (ไม่ใช่แค่ประมาณการ):** รัน verifier จริงบนข้อที่ WCB flag (target recall 1.00)
→ **F1 0.628 ± 0.006** (3 seeds) · 119 calls · $0.124
- **แย่กว่า baseline เอเจนต์เดี่ยว (0.690) และ *แพงกว่า* ด้วย ($0.124 vs $0.094)** แพ้ทุกแกน
- paired bootstrap vs baseline: ผลต่าง **-0.062**, 95% CI [-0.152, +0.025] · McNemar: base ถูก-cascade ผิด 14 / cascade ถูก-base ผิด 9
- (ตอนดูแค่ dry-run เคยเดาว่า F1 จะร่วงถึง ~0.42 เดาแรงเกินไป verifier ปัดตกได้เยอะกว่าที่คิด
  แต่ **ข้อสรุปไม่เปลี่ยน**: จ่ายแพงกว่า baseline เพื่อได้ F1 ต่ำกว่า)

*(หมายเหตุการทำซ้ำ: รอบนี้ WCB @0.5 ได้ F1 0.606 / 0.576 / 0.588 → mean **0.590** (SD 0.012)
ต่ำกว่าที่รายงานไว้ข้อ 5 (0.620 ± 0.005) และแกว่งกว่า น่าจะมาจาก torch/transformers คนละเวอร์ชัน
แต่แปลว่าตัวเลข WCB ในตารางอาจ "อ่อน" กว่าที่เห็น และ SD ±0.005 อาจต่ำเกินจริง)*

**7. เอเจนต์เดี่ยว + ปรับ threshold ≈ multi-agent v2 ที่ราคาครึ่งเดียว (finding ที่สั่นธีสิสที่สุด)**

ทุกระบบข้างบนตอบ 0/1 ตายตัว "ที่จุดทำงานจุดเดียว" แต่ GPT ไม่ได้ตอบ 0/1 จริงๆ มันตอบ *token* ที่มี logprob
ดึง P("1") ออกมาจาก logprob ของ token label ได้เลย (`gpt_threshold.py`) โดย **ไม่แตะ prompt และไม่เพิ่ม call แม้แต่ครั้งเดียว**
→ คะแนนนี้ได้มา "ฟรี" จาก call ที่จ่ายไปแล้วใน baseline

GPT **จัดอันดับเป็น** (ต่างจาก WCB ในข้อ 6 อย่างสิ้นเชิง) เลื่อน threshold แล้วซื้อ precision ได้จริง:

| tau | precision | recall | F1 |
|---|---|---|---|
| 0.50 (argmax = baseline เดิม) | 0.526 | **1.000** | 0.690 |
| 0.75 | 0.558 | 0.967 | 0.707 |
| 0.95 | 0.596 | 0.933 | 0.727 |
| 0.99 | 0.667 | 0.800 | 0.727 |

เลือก threshold แบบ **leave-fold-out** (tau ≈ 0.984 ทุก fold) → **F1 0.725 · prec 0.641 · recall 0.833 · 127 calls · $0.094**

**เทียบกับ v2 (2 agents) แบบ paired นี่คือคำถามเงินล้วนๆ ว่า "agent ที่ 2 คุ้มไหม":**
- v2 0.744 vs threshold 0.725 → ผลต่าง **+0.019**, 95% CI **[-0.076, +0.121]** → **คร่อม 0 กว้างมาก**
- โอกาสที่ v2 **ไม่ได้** ดีกว่า = **34%**
- McNemar: v2 ถูก-threshold ผิด **7** ข้อ | threshold ถูก-v2 ผิด **8** ข้อ → **เสมอกันสนิท**
- **แต่ v2 แพงกว่า 1.80× (+$0.075, +56 calls)**

=> **agent ตัวที่ 2 ซื้อ F1 ที่แยกจาก noise ไม่ออก ด้วยราคาเกือบเท่าตัว**
   สิ่งที่ v2 ทำได้จริง (ซื้อ precision โดยเสีย recall ให้น้อยที่สุด) **ปุ่ม threshold ตัวเดียวก็ทำได้เกือบเท่ากัน ฟรี**

**ข้อควรระวัง (อย่ารีบเปลี่ยนไปใช้ threshold):** โปรไฟล์ error ต่างกันชัด
- v2: recall **0.967** (พลาดประชด 1 ข้อ) ยังรักษาของดีจาก detector ไว้
- threshold: recall **0.833** (พลาดประชด 5 ข้อ) มันซื้อ precision ด้วยการ *ยอมทิ้ง recall*
→ ถ้างานคือ "คัดกรองห้ามพลาด" **v2 ยังชนะ** และ 0.075 ดอลลาร์นั้นคุ้ม
→ ถ้าวัดกันที่ F1 ล้วน **ปุ่มฟรีชนะขาดในเชิงราคา**

**บทเรียน (ต่อจากข้อ 4):** ก่อนจะถามว่า *"เพิ่ม agent ไหม"* ต้องถามก่อนว่า
***"รีดของที่ agent ตัวเดียวให้มาอยู่แล้ว ออกมาหมดหรือยัง"*** ที่นี่คำตอบคือ **ยัง**
baseline ทิ้ง logprob (ข้อมูลที่จ่ายเงินซื้อมาแล้ว) ลงถังขยะทุกข้อ แล้วเราไปซื้อ agent ตัวที่ 2 มาทดแทน
**การเทียบ multi-agent กับ baseline ที่ยังไม่ถูก tune ให้สุด = การเทียบที่ให้เครดิต multi-agent เกินจริง**

**8. การเทียบที่ยุติธรรม: เทียบทุกระบบกับ baseline ที่ tune แล้ว → ไม่มีระบบไหนชนะแบบมีนัยสำคัญเลย**

finding 1-7 เทียบทุกอย่างกับ baseline @argmax (0.690) ซึ่ง "ทิ้ง logprob" ที่จ่ายเงินซื้อมาแล้ว
= ให้ multi-agent สู้กับคู่ต่อสู้ที่ถูกมัดมือ คู่เทียบที่ถูกคือ **baseline+threshold (0.725, ราคาเท่าเดิม)**
รันใหม่ทั้งหมด (`compare_fair.py`, paired bootstrap, ไม่ยิง API ใช้ pred เดิม):

| ระบบ | F1 | ต่างจาก baseline+threshold | 95% CI | P(ไม่ดีกว่า) | McNemar | ราคา |
|---|---|---|---|---|---|---|
| conservative (v2) | 0.744 | +0.019 | [-0.073, +0.113] | 36% | 7-8 | $0.169 |
| v1aggressive | 0.714 | -0.010 | [-0.159, +0.135] | 57% | 11-8 | $0.157 |
| hybrid | 0.700 | -0.025 | [-0.117, +0.071] | 70% | 6-11 | $0.407 |
| debate | 0.694 | -0.030 | [-0.158, +0.096] | 69% | 9-12 | $0.695 |
| cascade | 0.628 | -0.097 | [-0.203, +0.012] | 96% | 7-20 | $0.124 |
| wangchanberta | 0.618 | -0.107 | [-0.235, +0.018] | 95% | 7-14 | $0.00 |

**ทุกระบบ CI คร่อม 0 ไม่มีระบบใดแยกออกจาก baseline+threshold ได้อย่างมีนัยสำคัญ**
ระบบ multi-agent ที่ดีที่สุด (v2) ยังมีโอกาสถึง 36% ที่ *ไม่ได้* ดีกว่าปุ่ม threshold ตัวเดียว ที่ราคา 1.80×
debate/hybrid ที่แพงกว่า 4-7× ก็อยู่ฝั่ง *ต่ำกว่า* ปุ่มฟรีด้วยซ้ำ

=> **ข้อได้เปรียบของ multi-agent v2 ที่เคยรายงานว่า "+0.054 vs baseline" หดเหลือ +0.019 vs baseline ที่ tune แล้ว
และหายไปใน 노이즈** ครึ่งหนึ่งของ +0.054 เดิม คือ "baseline ถูก handicap" ไม่ใช่ "multi-agent เก่ง"

**9. cost-quality frontier: เปลี่ยน "โมเดล" ขยับ cost ได้ 25× โดย F1 แทบไม่ขยับ และโมเดลถูกได้ F1 สูงสุด**

finding 1-8 อยู่บน gpt-4o โมเดลเดียว โปรเจกต์ชื่อ "Cost-Aware" จึงควรวัดแกน *โมเดล* ด้วย ไม่ใช่แค่แกนสถาปัตยกรรม
เอาสถาปัตยกรรมที่ดีสุด+ถูกสุด (เอเจนต์เดี่ยว + threshold, finding 7) ไปรันบน 5 โมเดลที่ราคาต่างกัน 25×
(`frontier.py`, threshold เลือกแบบ leave-fold-out เหมือนเดิม):

| โมเดล | F1 | prec | recall | $/รัน (127 ข้อ) | $/1M in |
|---|---|---|---|---|---|
| gpt-4.1-nano | 0.706 | 0.632 | 0.800 | $0.0038 | $0.10 |
| gpt-4o-mini | 0.676 | 0.568 | 0.833 | $0.0056 | $0.15 |
| **gpt-4.1-mini** ⭐ | **0.727** | 0.667 | 0.800 | $0.0150 | $0.40 |
| gpt-4.1 | 0.716 | 0.649 | 0.800 | $0.0752 | $2.00 |
| gpt-4o | 0.725 | 0.641 | 0.833 | $0.0940 | $2.50 |

- **F1 อยู่ในช่วง 0.68-0.73 ตลอดทั้งแถบราคา 25×** ช่วง 0.05 นี้เล็กกว่า CI ที่ n=30 (±0.10) → แทบทั้งหมดคือ noise
- **โมเดลที่ได้ F1 สูงสุดคือ gpt-4.1-mini ($0.015) ไม่ใช่ gpt-4o ($0.094)** โมเดลที่ทั้งโปรเจกต์สร้างอยู่บนนั้น ไม่ใช่จุดที่ดีที่สุดด้วยซ้ำ
- gpt-4o-mini เป็นตัวหลุด (F1 ต่ำสุด 0.676) → F1 ไม่ได้ไล่ตามราคาแบบเส้นตรง เลือกโมเดลถูกต้องลองจริง ไม่ใช่เดาจากราคา

**หมัดเด็ด เอเจนต์เดี่ยวโมเดลถูก vs multi-agent เรือธง (paired, n=127):**

| | F1 | ราคา | agents / calls |
|---|---|---|---|
| gpt-4.1-mini + threshold | 0.727 | **$0.015** | 1 agent / 127 |
| gpt-4o v2 (multi-agent เรือธง) | 0.744 | $0.169 | 2 agents / 183 |

ผลต่าง +0.016, 95% CI **[-0.094, +0.135]** (คร่อม 0) · P(v2 ไม่ดีกว่า) = **39%** · **v2 แพงกว่า 11×**

=> ระบบ multi-agent ที่ดีที่สุดของโปรเจกต์ **แยกจากเอเจนต์เดี่ยวบนโมเดลถูกไม่ออก แต่แพงกว่า 11 เท่า**
   "cost-aware" ตัวจริงของงานนี้ไม่ใช่ "เลือกสถาปัตยกรรม" แต่คือ **"เลือกโมเดลถูก + อ่าน logprob ของมัน"**

*(ราคาเป็นค่าประมาณใน `frontier.py`/PRICE ต้องตรวจกับราคาปัจจุบัน; token counts วัดจริงจาก API เป๊ะ 34,535 in / 762 out เท่ากันทุกโมเดล)*

**10. แก้ precision ที่ต้นทาง (few-shot ใน detector) ตัด FP ได้จริง แต่ F1 ไม่ขยับอย่างมีนัยสำคัญ**

finding 1 ชี้ว่าคอขวดคือ precision: FP 27 ข้อ ส่วนใหญ่คือ "รีวิวสมดุล" (ชมจริง+ติจริง) ที่ LLM อ่านเป็นประชด
ทุกระบบก่อนหน้าล้าง FP นี้ *หลัง* detector ยิง finding นี้ลองแก้ที่ **ตัว detector เอง**:
ใส่กฎ "การเสแสร้ง" ชัดๆ + few-shot **สังเคราะห์** 4 ตัวอย่าง (ไม่ใช่ gold กัน leak) สอนเส้นแบ่ง "รีวิวสมดุล = 0"
รันบน gpt-4o + threshold leave-fold-out (ต่างจาก finding 7 แค่ prompt):

| | F1 | precision | recall | FP | FN |
|---|---|---|---|---|---|
| baseline @argmax (prompt เดิม) | 0.690 | 0.526 | 1.000 | **27** | 0 |
| + threshold (prompt เดิม) | 0.725 | 0.641 | 0.833 | 19 | 5 |
| **+ threshold + few-shot สมดุล** | 0.738 | **0.686** | 0.800 | **11** | 6 |

- **กลไกทำงานจริง: FP ร่วง 27 → 11 · precision 0.526 → 0.686**, few-shot ตัด "รีวิวสมดุล" ออกได้ตามตั้งใจ
- **แต่ F1 ขยับแค่ 0.725 → 0.738 = ไม่มีนัยสำคัญ** (paired vs prompt เดิม: +0.014, 95% CI [-0.075, +0.100], P(ไม่ดีกว่า) 37%)
  เพราะการตัด FP แลกมาด้วย recall ที่ลดลง (0.833 → 0.800, FN 5 → 6) ได้อย่างเสียอย่าง
- **เทียบ v2 multi-agent (paired): 0.738 vs 0.744 → ต่าง +0.005, P(v2 ไม่ดีกว่า) 46% = โยนหัวก้อย**
  → **detector เดี่ยว + prompt ดี = multi-agent 2 agent** (อีกหนึ่งเส้นหลักฐานว่าไม่ต้องเพิ่ม agent)

**ข้อควรระวังเรื่อง cost (แก้ความเข้าใจผิดของตัวเอง):** few-shot ทำ prompt ยาวขึ้น → input token ~2× → **$0.19 ไม่ใช่ $0.094**
"ฟรี" เฉพาะ *จำนวน call* (ยัง 127) ไม่ใช่ *ค่าเงิน* ที่ราคานี้พอๆ กับ v2 ($0.169) ไม่ได้ถูกกว่า
(ตัวถูกจริงยังเป็น gpt-4.1-mini prompt เดิม $0.015 จาก finding 9)

**สรุป finding นี้:** ทุกอย่าง baseline, threshold, few-shot, v2, debate ตกอยู่ในแถบ F1 0.69-0.74 ที่ n=30 แยกกันไม่ออก
แม้แก้ที่คอขวด (precision) ตรงๆ ก็ยังชนแถบเดียวกัน → ตอกย้ำว่า **เพดานคือข้อมูล ไม่ใช่วิธี**

**11. ขยาย gold เป็น 45 ประชด (จาก 30) แล้วเทียบใหม่ ข้อสรุป "แยกไม่ออก" ยังยืน (แข็งขึ้น)**

finding 6-10 ทุกข้อชนเพดานเดียวกัน: n(ประชด)=30 → CI กว้างเกินฟันธง จึงลอง "ยกเพดาน" ด้วยการ label เพิ่ม
คน label 37 ข้อบนสุดของ `to_label_next.csv` (เลือกด้วย logprob P>0.2) → ได้ 15 ประชด + 15 ไม่ + 7 ไม่แน่ใจ
→ **gold ขยาย 127→157 ข้อ, ประชด 30→45 (+50%)** เก็บเป็น `gold_expanded.csv` (ไม่แตะ gold.csv canonical)

เทียบคู่หลัก (baseline+threshold vs v2 multi-agent) บน gpt-4o, threshold leave-fold-out, บนเซ็ต 157:

| | n=127 (ประชด 30) | n=157 (ประชด 45) |
|---|---|---|
| baseline+threshold F1 | 0.725 | 0.745 |
| v2 multi-agent F1 | 0.744 | 0.724 |
| ต่าง (v2−thr) | +0.019 | **−0.021** |
| 95% CI | [−0.073, +0.113] | [−0.112, +0.072] |
| P(v2 ไม่ดีกว่า) | 36% | **68%** |
| McNemar (v2 : thr) | 7 : 8 | 11 : 17 |

- **ยังคร่อม 0 เพิ่มประชด 50% ก็ยังแยก multi-agent จากเอเจนต์เดี่ยวไม่ออก** ข้อสรุป finding 8 ยืน
- point estimate **พลิกไปข้าง threshold** (−0.021) แต่ **อย่าอ่านว่า "threshold ชนะ"** เพราะ...

**อคติที่ต้องประกาศ (สำคัญ):** 30 ข้อใหม่ **ไม่ได้สุ่ม** เลือกด้วยคะแนน logprob ของโมเดล (P>0.2)
→ เอนเข้าหาระบบ logprob-threshold โดยตรง · absolute F1 บน 157 จึง **เทียบกับ 127 เดิมไม่ได้**
อ่านได้แค่ทิศ: "multi-agent ยังไม่โผล่ว่าดีกว่า แม้ข้อมูลเยอะขึ้น"

**บทเรียนเรื่องกำลังสถิติ:** CI แทบไม่แคบลง (0.186 → 0.184) ทั้งที่เพิ่มประชด 50%
เพราะข้อที่เลือกมาคือ "ข้อยาก/ก้ำกึ่ง" (P 0.2-0.8 + ที่โมเดลเดาผิด) ข้อพวกนี้ noisy ไม่ได้เพิ่ม power มากอย่างที่หวัง
→ **ถ้าอยากได้ CI แคบจริง ต้องสุ่มประชดเพิ่ม (ไม่ใช่เลือกเฉพาะข้อยาก) และต้องเยอะกว่านี้มาก**

**12. ทดสอบข้ามโดเมนจริง (Pantip): precision พังเพราะจับคำชมจริงเป็นประชด**

finding 1-11 ทุกอย่างวัดบน gold โดเมนเดียว (รีวิว Wongnai + ทวีต Wisesight) จึงทดสอบจริงว่าข้ามโดเมนได้ไหม
คน label คอมเมนต์ Pantip 55 ข้อด้วยมือ (คนตัดสินเอง ไม่ใช่โมเดล) จากกระทู้การเมือง + กระทู้อาหาร/รีวิว
วัดด้วย gpt-4.1-mini (balanced) ผ่าน `validate_link.py` + `eval_domain.py`:

| | gold เดิม (รีวิว/ทวีต) | Pantip (n=55) |
|---|---|---|
| precision | 0.68 | **0.40** |
| recall | 0.83 | 0.86 |
| F1 | 0.75 | **0.545** |

TP 6 · FP 9 · FN 1 · TN 39 · F1 95% CI [0.25, 0.77]

**หัวใจ: precision ร่วง 0.68 -> 0.40** เพราะ FP 9 ข้อ คือ "คำชม/ข้อความปกติที่โมเดลจับเป็นประชด"
recall ยังสูง (0.86 จับประชดจริงได้ 6/7) แปลว่าโมเดล "เห่าหมาป่า" นอกโดเมน: ไม่พลาดประชด
แต่จับคำชมจริงเป็นประชดเยอะ ยืนยันสิ่งที่เห็นตอนลอง YouTube (over-flag คำชม) ด้วยตัวเลขจริง

**บทเรียนเรื่องตัวอย่างเล็กหลอกได้:** pilot รอบแรก 14 ข้อ (กระทู้การเมืองล้วน) ได้ F1 0.80 ดูเหมือนผ่าน
แต่เพราะมีคำชมจริงน้อย พอเติมคำชม 41 ข้อจากกระทู้อาหาร precision เหลือ 0.40 -> ตัวอย่างน้อยสรุปผิดได้

ความหนักแน่น: finding ฝั่ง precision แน่น (FP 9 จาก ~48 negative คือ pattern ชัด ไม่ใช่ noise)
ฝั่ง recall/F1 ยังกำลังน้อย (ประชดแค่ 7 CI กว้าง) จึงอ่าน F1 เป็นทิศทาง แต่ "over-flag คำชม" ฟันธงได้

**สรุป: โมเดลไม่ทรานส์เฟอร์ข้ามโดเมนดี** ใช้ได้เฉพาะเนื้อหาคล้าย gold (รีวิว/โพสต์สั้น)
โดเมนอื่นต้อง label + tune ใหม่ (ปุ่มแก้ few-shot ในเว็บช่วยลด FP ตรงนี้ได้) นี่คือความเสี่ยงใหญ่สุดตอน deploy จริง

**13. open-model frontier (แก้ใหม่): โมเดลเปิดแพ้จริง แต่ช่องว่างเล็กกว่าที่เคยรายงาน ~40% และ "โมเดลไทยเฉพาะทาง" ไม่ได้เปรียบเลย**

ต่อยอดแกน cost ตรงๆ: *"F1 เท่าไรที่ซื้อได้ด้วย API $0"* รันโมเดลเปิดในเครื่อง (RTX 3060 Ti, 4-bit)

**สิ่งที่ผิดในเวอร์ชันแรกของ finding นี้ (แก้แล้ว 2 จุด):**

1. **baseline GPT ผิด** เวอร์ชันแรกเขียน GPT bot = F1 0.696 แต่ finding 9 (ตาราง `frontier.py`)
   ระบุ gpt-4.1-mini = **0.727** (prec 0.667 / recall 0.800) รันซ้ำจาก `frontier_probs_gpt-4.1-mini.csv`
   ได้ 0.727/0.667/0.800 ตรงเป๊ะกับ finding 9 และไม่มีไฟล์ไหนในโปรเจกต์ให้ค่า 0.696 เลย -> 0.696 คือตัวเลขผิด
2. **โปรโตคอลไม่แฟร์** เวอร์ชันแรกเทียบ *GPT ที่ tune threshold แล้ว* กับ *โมเดลเปิดแบบ argmax ดิบ*
   ซึ่งเป็นความผิดพลาดแบบเดียวกับที่ finding 8 เคยจับได้ ("baseline ถูก handicap")
   -> รอบนี้ให้ทั้งสองฝั่งใช้ leave-fold-out (StratifiedKFold 5, seed 42) เหมือนกันหมด

**ผลข้างเคียง: dynamic few-shot (RAG) ไม่น่าจะได้ผลกับงานนี้ วัดแล้ว ฟรี**

ไอเดีย "ดึงตัวอย่างที่คล้ายที่สุด 2 ข้อมาใส่ prompt แทน few-shot ตายตัว" ต้องอาศัยสมมติฐานว่า
*ข้อความที่คล้ายกันเชิงความหมาย = ป้ายกำกับเหมือนกัน* วัดสมมติฐานนี้ตรงๆ บน 3 encoder:

| encoder | เพื่อนบ้าน top-2 ป้ายตรงกับ query | kNN F1 | **gap** (ประชด↔ประชด ลบ ประชด↔ไม่ประชด) |
|---|---|---|---|
| WangchanBERTa | 0.787 | 0.481 | **-0.058** |
| BGE-M3 | 0.772 | 0.528 | **-0.001** |
| multilingual-e5-large | 0.787 | 0.588 | **-0.003** |

(chance = 0.639 · gap > 0 แปลว่าข้อประชดเกาะกลุ่มกัน)

**gap ติดลบทั้ง 3 ตัว** ข้อประชด **ไม่ได้** อยู่ใกล้กันเองมากกว่าอยู่ใกล้ข้อไม่ประชด
encoder ที่แรงขึ้นดัน gap จาก -0.058 มาที่ ~0 (คือ "ไม่มีข้อมูล") ไม่ใช่เป็นบวก
-> ดึง top-2 ที่คล้ายที่สุดมา จะได้ตัวอย่างที่ป้ายกำกับ **แทบไม่สัมพันธ์** กับคำตอบของข้อที่กำลังตัดสิน
สำหรับข้อ *ประชด* (คลาสน้อย ที่ recall อ่อนอยู่แล้ว) จึงเสี่ยงได้ตัวอย่าง "ไม่ประชด" มาสอนผิดทาง

ประกอบกับ finding 10 (few-shot สังเคราะห์แบบสมดุลที่ทำมือ ได้ +0.014 ไม่มีนัยสำคัญ ที่ราคา 2 เท่า)
-> **ไม่คุ้มที่จะลงแรง/ลงเงินกับ RAG few-shot บนชุดข้อมูลนี้** (บันทึกไว้ว่า "ทดสอบแล้ว ไม่ใช่ยังไม่ได้ลอง")
*(หมายเหตุที่ยังค้าง: gap รวมทั้งชุด ~0 แต่ kNN k=1 ยังได้ 0.588 -> สัญญาณอยู่ในย่านใกล้ๆ เฉพาะที่
ไม่ใช่โครงสร้างระดับคลาส ยังไม่ได้เจาะว่าทำไม)*

**ผลข้างเคียงที่ 2: semantic cache (แคชด้วยความคล้ายเชิงความหมาย) ใช้ไม่ได้กับงานนี้ วัดแล้ว ฟรี**

ไอเดีย "ถ้าประโยคใหม่คล้ายของเก่า 95% ให้คืนคำตอบเดิม ไม่ต้องยิง LLM" อาศัยสมมติฐานเดียวกับ RAG:
*ข้อความคล้ายกัน = คำตอบเหมือนกัน* วัดตรงๆ บนคู่ข้อความทั้งหมด 8,001 คู่ (encoder: multilingual-e5-large):

| เกณฑ์ความคล้าย | จำนวนคู่ที่เข้าเกณฑ์ | % ของคู่ทั้งหมด | คืนคำตอบ **ผิด** |
|---|---|---|---|
| 0.95 - 0.99 | **0** | 0.0% | (cache ไม่เคยทำงาน) |
| 0.93 | 27 | 0.3% | 3.7% |
| 0.90 | 618 | 7.7% | **15.2%** |

**ความคล้ายสูงสุดในชุดข้อมูลคือ 0.942** ไม่มีคู่ไหนถึง 0.95 เลย
-> ตั้ง threshold ที่ 95% = cache **hit rate 0%** (เขียนโค้ดไปก็ไม่มีอะไรวิ่งผ่าน)
-> ลด threshold ให้ทำงานได้ ก็เริ่มคืนคำตอบผิด: ที่ 0.90 ได้ hit 7.7% แต่ **ผิด 15.2%**
(สุ่มมา 2 ข้อมั่วๆ ป้ายต่างกัน 36.4% -> ดีกว่าสุ่ม แต่ยังห่างไกลจาก "ปลอดภัย")

**คู่ที่คล้ายที่สุดแต่ป้ายต่างกัน 5 อันดับแรก = รีวิวร้านอาหารทั้งหมด ที่ไปคล้ายกับรีวิวประชดอันเดียวกัน**
คล้ายเพราะ *หัวข้อเดียวกัน* ไม่ใช่เพราะ *ท่าทีเดียวกัน* กลไกเดียวกับผล gap ข้างบน
สมมติฐาน "คล้าย = คำตอบเหมือน" จึงผิดกับงานประชดโดยเฉพาะ เพราะประชดนิยามด้วย *เจตนาที่สวนกับเนื้อความผิวเผิน*
ซึ่งเนื้อความผิวเผินคือสิ่งเดียวที่ embedding วัด · (และประหยัดได้แค่ ~$0.0001 ต่อ call ที่แคชได้)

finding 12 แสดงแล้วว่า precision ข้ามโดเมนพัง 0.68 -> 0.40 และ cue ระดับคำ **ควรจะทรานส์เฟอร์แย่กว่าโมเดล ไม่ใช่ดีกว่า**
-> ข้อสรุปนี้จำกัดอยู่ที่ *โดเมนนี้* เท่านั้น **ยังไม่ได้วัดบน Pantip** (ชุด `domain_*_labeled.csv` อยู่ใน .gitignore

**15. Micro-router 3 ทาง: จ่าย 40% ของราคาเต็ม ได้ F1 95% ของ GPT bot**

finding 14 ตั้ง "พื้น $0" ไว้ที่ ~0.59 ส่วน GPT bot อยู่ที่ 0.727 คำถามที่เหลือคือ
**ต้องจ่ายเท่าไหร่ถึงจะไต่ระหว่างสองจุดนั้น** ไม่ใช่ "จ่ายหมดหรือไม่จ่ายเลย"

`router.py` ใช้ threshold **สองตัว** (cascade.py ใช้ตัวเดียว) แบ่งเป็น 3 ทาง:
prob < lo → ตัดสินเอง "ไม่ประชด" (ฟรี) · prob ≥ hi → ตัดสินเอง "ประชด" (ฟรี) ·
ระหว่างกลาง → **ไม่แน่ใจ** → ส่งให้ GPT (จ่ายเฉพาะข้อพวกนี้)
กวาด escalation budget b = สัดส่วนข้อที่ยอมส่งต่อ (uncertainty sampling รอบ tau) แบบ leave-fold-out
3 seeds (42/7/2024) ข้อใน fold k ใช้ threshold จากอีก 4 folds เท่านั้น

| budget | escalate | F1 | prec | recall | calls | cost | % ของ GPT bot |
|---|---|---|---|---|---|---|---|
| 0.00 | 0 (0%) | 0.590 | 0.537 | 0.656 | 0 | **$0** | 81% |
| 0.10 | 15 (12%) | 0.629 | 0.579 | 0.689 | 15 | $0.016 | 87% |
| 0.20 | 27 (21%) | 0.650 | 0.591 | 0.722 | 27 | $0.028 | 89% |
| 0.30 | 40 (31%) | 0.666 | 0.619 | 0.722 | 40 | $0.042 | 92% |
| **0.40** | **51 (40%)** | **0.690** | 0.644 | 0.744 | 51 | **$0.053** | **95%** |
| 0.70 | 89 (70%) | 0.694 | 0.650 | 0.744 | 89 | $0.094 | 95% |
| 1.00 | 124 (98%) | 0.721 | 0.663 | 0.789 | 124 | $0.130 | 99% |

=> **จ่าย 40% ของราคา ได้ F1 0.690 = 95% ของ GPT bot** · เลย 0.40 ไปแล้วเส้นแบน

b=1.00 ได้ 0.721 เทียบกับ GPT bot 0.727 ที่รายงานไว้ -> ปลายทางตรงกัน = ท่อ leave-fold-out ถูก

**ผลพลอยได้ที่สำคัญกว่าตัว router: การจูน threshold ของ WCB แบบ leave-fold-out *ทำให้แย่ลง***

| วิธีตั้ง threshold | F1 (เฉลี่ย 3 seeds) | ใช้รายงานได้ไหม |
|---|---|---|
| จูนบน gold ทั้งชุด แล้ววัดบน gold ชุดเดิม | **0.627** | **ไม่ได้ leak** |
| จูนแบบ leave-fold-out (จากอีก 4 folds) | 0.556 | ได้ |
| **ไม่จูนเลย ใช้ 0.5 ตรงๆ** | **0.590** | ได้ |

ที่ n=127 (ประชด 30) การจูน threshold บน 4 folds แล้วเอาไปใช้กับ fold ที่ 5 **แพ้การไม่จูนเลย 0.034**
ส่วนช่องว่างระหว่าง 0.627 (leak) กับ 0.556 (ซื่อสัตย์) = **0.071 คือภาพลวงล้วนๆ**
-> `router.py` จึงดีฟอลต์เป็น `--tau-mode fixed` (0.5) · `--tau-mode tuned` ไว้ให้ทำซ้ำผลนี้ได้
กลไกเดียวกับ finding 13 (40% ของ gap เดิมเป็น measurement error) **ปุ่มจูนที่ n น้อยคือหนี้ ไม่ใช่ทรัพย์สิน**

และ WCB ที่ 0.5 ได้ 0.590 ซึ่ง**ตรงกับ regex `555` (0.590) และ kNN e5 (0.588) พอดี**
-> วิธีฟรี 3 วิธีที่ไม่เกี่ยวกันเลย (lexical / embedding / neural) จบที่ 0.59 เท่ากันหมด
ยืนยันว่า "ชั้น $0 ที่ ~0.59" ของ finding 14 เป็นเพดานจริงของงานนี้ ไม่ใช่ความบังเอิญของวิธีใดวิธีหนึ่ง

**สาเหตุอยู่ที่ตัวงาน ไม่ใช่ตัวโค้ด** แคชเชิงความหมายตั้งบนสมมติฐาน "คล้าย = คำตอบเหมือน"
แต่ประชดนิยามด้วย *เจตนาที่สวนกับเนื้อความผิวเผิน* ซึ่งเนื้อความผิวเผินคือสิ่งเดียวที่ embedding วัด
-> รีวิวร้านอาหาร 2 อันคล้ายกันเพราะ **หัวข้อเดียวกัน** ไม่ใช่ **ท่าทีเดียวกัน**
บันทึกว่า "สร้างแล้ว วัดแล้ว ไม่เวิร์ค เพราะเหตุผลนี้" ซึ่งหนักแน่นกว่า "ยังไม่ได้ลอง"

**17. async debate: ตัด latency 54% โดยราคาไม่ขยับเลย (วัดจริงแล้ว)**

`multiagent_debate.py` รัน อัยการ → ทนาย → ผู้พิพากษา เรียงกัน แต่**อัยการกับทนายไม่ได้อ่านคำแถลงของกันและกัน**
(ดู `_argue`: user prompt คือ `"ข้อความ: {text}"` เท่านั้น) -> สองตัวนี้อิสระต่อกันจริง ยิงพร้อมกันได้
`async_debate.py` ใช้ `asyncio.gather` -> latency = max(อัยการ, ทนาย) + ผู้พิพากษา แทนผลรวม

**รันจริงแล้ว 30 ข้อ (gpt-4o, concurrency 4, $0.165, error 0)** เทียบกับ sequential บน **30 ข้อชุดเดียวกัน**:

| | p50 | p95 | หมายเหตุ |
|---|---|---|---|
| sequential | 4583 ms | 5643 ms | อัยการ -> ทนาย -> ผู้พิพากษา |
| **async** | **2100 ms** | **3692 ms** | (อัยการ \|\| ทนาย) -> ผู้พิพากษา |
| **ลดลง** | **54%** | 35% | |

- ขั้นโต้แย้ง (อัยการ‖ทนาย) p50 **1471 ms** = max ของสองฝ่าย ไม่ใช่ผลรวม -> ยืนยันว่า gather ทำงานจริง
- **จำนวน call/token เท่าเดิมเป๊ะ** (3 calls/ข้อ · $0.165/30 ข้อ = $0.0055/ข้อ ตรงกับ $0.695/127 ข้อ)
- throughput: wall-clock 20s ต่อ 30 ข้อ = **0.65 s/ข้อ** ที่ concurrency 4 (sequential 4.58 s/ข้อ) = เร็วขึ้น ~7 เท่า
- error 0 ที่ concurrency 4 -> ไม่ชน rate limit ที่ระดับนี้

**ลดได้ 54% ไม่ใช่ 33% ตามที่คาดไว้ตอนแรก** เพราะตอนคาดคิดว่า 3 ด่านราคาเท่ากัน แต่จริงๆ
ด่านผู้พิพากษา `max_tokens=20` ส่วนด่านโต้แย้ง `max_tokens=120` -> การดึง *ด่านโต้แย้งเต็มๆ ด่านหนึ่ง*
ออกจากเส้นทางวิกฤต ประหยัดได้มากกว่า 1 ใน 3 · บทเรียน: **ประเมิน latency ต้องดู max_tokens ของแต่ละด่าน
ไม่ใช่นับจำนวนด่าน**

⚠️ **F1 ของรันนี้ (0.400) ใช้เทียบอะไรไม่ได้** 30 ข้อแรกของ gold มีข้อประชดแค่ **2 ข้อ**
(gold.csv ไม่ได้สลับลำดับ ข้อประชดไปกระจุกอยู่ท้ายไฟล์) -> TP 1 / FN 1 / FP 2 คือ noise ล้วนๆ
รันนี้ตั้งใจวัด **latency** ซึ่งเป็นสมบัติราย*ข้อ* จึงพอที่ n=30 · **ห้ามเอา F1 ตัวนี้ไปรายงาน**

**และ "เช็คว่า async ให้ผลเหมือน sequential" ทำไม่ได้ด้วยการรันซ้ำอยู่ดี** ไม่มีการ pin temperature
ที่ไหนเลย (default 1.0) แปลว่า sequential เองก็ทำซ้ำตัวเองไม่ได้ ผลต่างที่เห็นจะเป็น sampling noise
ไม่ใช่ผลของสถาปัตยกรรม -> ถ้าอยากยืนยันจริงต้อง `temperature=0` ทั้งสองฝั่งก่อน แล้วค่อยเทียบ
(ดูหัวข้อ "ข้อควรระวังตอนเขียนรายงาน")

ราคาถ้ารันครบ 127 ข้อ = **~$0.695** (gpt-4o, วัดจาก `multiagent_preds_gpt_debate.csv`:
183,661 in / 23,585 out) *ไม่ใช่* $0.169 ที่ multiagent_debate.py พิมพ์เทียบไว้ ตัวนั้นคือ pipeline v2

**18. load test: p50 = 28 ms เพราะ 83% ของ traffic ไม่แตะ LLM เลย**

`loadtest.py` ยิง request แบบ Poisson ผ่าน router (mock latency จากค่าที่วัดจริง: WCB 26ms /
GPT call 751ms / debate 4557ms) โหมด `--mock` ฟรี ใช้จูน concurrency ได้ไม่จำกัดรอบ

ที่ rps 8 / budget 0.20: escalate 17% · **p50 28 ms** · p95 923 ms · $0.088/นาที · error 0
แยกตามเส้นทาง: auto n=137 p50 **26 ms** (ฟรี) vs escalated n=28 p50 **840 ms** (ยิง LLM)
-> **ช่องว่าง 32 เท่าระหว่างสองเส้นทางคือ "ราคาของความไม่แน่ใจ"** และ p50 ของทั้งระบบเกาะเส้นทางฟรี

sweep concurrency ที่ rps 25 (12s) เจอคอขวดชัด:

**ข้อจำกัดที่ต้องพูดตรงๆ ในรายงาน:** gold มี 127 ข้อ นี่ไม่ใช่ traffic ระดับโปรดักชันจริง
สคริปต์นี้ตอบได้แค่ "orchestration ถูกไหม / p95 เท่าไร / cost ต่อนาทีเท่าไร"

**19. ขยาย gold จนครบกอง harvest (595 ข้อ, ประชด 104): ธีสิสยืนบนข้อมูล 4.7 เท่า และ "อัปเกรดฟรี" กลายเป็นมีนัยสำคัญครั้งแรก**

finding 11 ทิ้งท้ายว่า "อยากได้ CI แคบจริง ต้อง label เยอะกว่านี้มาก" รอบนี้ทำจนจบกอง:
ติดป้าย `harvest_to_review.csv` ครบ **495/495** ผ่าน `label_ui.py` (blind เหมือนเดิม)
→ **`gold_v2.csv` 595 ข้อ, ประชด 104 (17.5%)** แล้วรันใหม่ทั้ง 3 ระบบครบทุกข้อ → `v2_results/`

| ระบบ (บน gold_v2, n=595) | F1 | prec | recall | calls | cost |
|---|---|---|---|---|---|
| baseline (argmax) | 0.337 | 0.206 | 0.933 | 595 | $0.275 |
| **baseline + threshold (leave-fold-out)** ⭐ | **0.377** | 0.238 | 0.894 | 595 | **$0.275** |
| v2 multi-agent (screener→verifier) | 0.357 | 0.222 | 0.913 | 1080 | $0.814 (2.96×) |

**19.1 ผลใหม่ที่สำคัญที่สุด: threshold ชนะ baseline แบบมีนัยสำคัญแล้ว (ครั้งแรกในโปรเจกต์)**

paired bootstrap n=595: threshold−baseline = **+0.040, 95% CI [+0.018, +0.060]**
→ **CI ไม่คร่อม 0** · McNemar **80 : 6** (เอียงหนักไปทาง threshold)

เทียบวิวัฒนาการของข้อนี้: n=127 → +0.035 (คร่อม 0) · n=302 → +0.015 CI [−0.033,+0.061] (คร่อม 0)
· **n=595 → +0.040 CI [+0.018,+0.060] (ไม่คร่อม)**
=> finding 1 ("อ่าน logprob ที่จ่ายเงินไปแล้ว = อัปเกรดฟรี") เลื่อนสถานะจาก *แนวโน้ม* เป็น
**ข้อสรุปที่ข้อมูลยืนยันได้** ที่ราคาเท่าเดิมเป๊ะ ($0.275 ทั้งคู่ · calls เท่ากัน 595)

**19.2 ธีสิสหลักยังยืน และ point estimate พลิกไปข้าง single agent ชัดขึ้น**

เทียบคู่ที่ตอบธีสิสจริง (multi-agent vs **เส้นฐานที่ยุติธรรม** = threshold ไม่ใช่ argmax):

| n | multi-agent − threshold | 95% CI | ความกว้าง CI | P(multi ไม่ดีกว่า) | McNemar (multi : thr) |
|---|---|---|---|---|---|
| 127 | +0.019 | [−0.073, +0.113] | 0.186 | 36% | 7 : 8 |
| 302 | +0.002 | [−0.044, +0.050] | 0.094 | 48% | 17 : 42 |
| **595** | **−0.019** | **[−0.047, +0.009]** | **0.056** | **91%** | **49 : 83** |

- **ยังคร่อม 0 → ยังฟันธงไม่ได้ว่า multi-agent แย่กว่า** แต่ที่ n=595 ทิศชัดขึ้นมาก:
  โอกาสที่ multi-agent *ไม่* ดีกว่า = **91%** และ McNemar เอียงไปทาง single agent เกือบ 2 เท่า
- จ่ายแพงกว่า **2.96×** (1080 calls vs 595) เพื่อผลที่ point estimate ติดลบ
=> ข้อสรุปธีสิสไม่เปลี่ยน แต่แข็งขึ้น: **ไม่มีหลักฐานว่า multi-agent คุ้ม และหลักฐานเริ่มเอียงไปทางไม่คุ้ม**

**19.3 hard negatives กดทุกสถาปัตยกรรมเท่ากัน (absolute F1 ร่วงต่อ)**

absolute F1 ร่วงจาก ~0.45 (n=302) เหลือ ~0.35 (n=595) เพราะข้อที่เพิ่มมาเป็น hard negative ล้วน:
กอง harvest ถูก keyword+LLM คัดว่า "หน้าเหมือนประชด" แต่คนตัดสินว่าไม่ใช่ถึง **403/495 (81%)**
→ สัดส่วนประชดเจือจางลง (22% → 17.5%) และ FP พุ่งทุกระบบพอๆ กัน (threshold: FP 297)
**precision พังเท่ากันหมด เพิ่ม agent ไม่ช่วย** — นี่คือคุณสมบัติของ *ข้อมูล* ไม่ใช่ของสถาปัตยกรรม

GPT ยัง "มั่นใจเกิน" หนักขึ้น: prob สุดขั้ว (<0.01 หรือ >0.99) **404/595 ข้อ (68%)**
แต่ที่น่าสนใจคือ **ต่อให้มวลเหลือน้อย threshold ก็ยังดึงกำไรได้อย่างมีนัยสำคัญ** (19.1)
· จูนแบบ leak บนทั้งชุดได้ 0.381 เทียบกับ leave-fold-out 0.377 → ช่องว่างแค่ 0.004
(เทียบกับ finding 15 ที่ WCB มีช่องว่าง leak 0.071) แปลว่า **threshold ของ GPT เสถียร ไม่ overfit**

**อคติที่ต้องประกาศ (ย้ำจาก finding 11):** gold_v2 **ไม่ใช่** ตัวแทน i.i.d. ของ traffic จริง
negative ถูก pre-select ให้ยาก → **absolute F1 ข้ามเซ็ตเทียบกันไม่ได้** อ่านได้เฉพาะ paired
ภายในเซ็ตเดียวกัน · `Gold/random_to_label.csv` (สุ่มจริง 250 ข้อ จาก pool 72,865) เตรียมไว้แล้ว
สำหรับทำ absolute number ที่อ่านได้ — **ยังไม่ได้ติดป้าย**
มุมกลับที่มีประโยชน์: gold_v2 ใช้เป็น **hard set** วัดความทน hard-negative ของระบบใดๆ ได้ทันที

**20. ทดสอบบนชุดสุ่มจริง (245 ข้อ, ประชด 4.9%): ตัวเลขเด่นทั้งหมดของโปรเจกต์เป็นภาพลวงของชุด mined และ "อ่าน confidence" คือสิ่งเดียวที่ทำให้ระบบใช้ได้จริง**

gold ทุกชุดก่อนหน้า (127 / 302 / 595) ถูก **pre-select ด้วย keyword+LLM** ให้หนาแน่นประชด (17–24%)
→ absolute F1 อ่านข้ามโลกจริงไม่ได้ (ประกาศไว้ตั้งแต่ finding 11) รอบนี้ปิดช่องนั้น:
สุ่มจริง 250 ข้อจาก pool 72,865 (seed 42) ติดป้าย blind ครบ → **base rate จริง = 4.9% (12/245)**
เทียบกอง harvest ~22% → **keyword filter ดันความหนาแน่นประชดขึ้น ~4.5 เท่า**

| ระบบ (บนชุดสุ่มจริง n=245, ประชด 12) | F1 | prec | recall | acc | flag rate |
|---|---|---|---|---|---|
| เดา "ไม่ประชด" เสมอ | 0.000 | — | 0.000 | **0.951** | 0% |
| cue เชิงคำ (ตัวฟรีที่ deploy จริง) | 0.090 | 0.055 | 0.250 | 0.751 | 22% |
| GPT-4o baseline (argmax) | 0.202 | 0.115 | 0.833 | 0.678 | **36%** |
| **GPT-4o + threshold (leave-fold-out)** ⭐ | **0.462** | 0.429 | 0.500 | 0.943 | 6% |

(95% CI ของ threshold F1 = [0.182, 0.688] — กว้างเพราะมีประชดแค่ 12 ข้อ ตัวเลข absolute อ่านหยาบๆ ได้ แต่ทิศชัด)

**20.1 "ชั้น $0 ที่ F1 0.59" เป็นภาพลวงของชุด mined — บนข้อมูลจริงมันเหลือ 0.09**

finding 14 เฉลิมฉลองว่า regex/cue ฟรีได้ F1 ~0.59 ("ชั้นล่างสุดของแกน cost")
แต่นั่นวัดบนชุดที่ประชด 24% **บนชุดสุ่มจริง 4.9% ตัว cue ร่วงเหลือ F1 0.090** (prec 0.055)
เพราะ cue เดียวกันที่ยิงโดนประชดในชุด mined ตอนนี้ยิงโดน "คนพูดปกติ" เต็มไปหมด (FP 52 ข้อ)
=> **ไม่ใช่คุณสมบัติของวิธี แต่เป็นคณิตศาสตร์ของ rare class**: precision ตายเมื่อ positive หายาก
"ชั้น $0" จริงๆ ของงานนี้บนโลกจริง = **แทบเท่าเดา "ไม่ประชด" เสมอ (F1 0, acc 95%)**

**20.2 GPT argmax ก็พังด้วยเหตุเดียวกัน — เหวี่ยงแหกว้างจน precision 0.115**

GPT-4o แบบ argmax (เจ้าเดียวกับที่ได้ F1 0.69 บนชุด mined) **ธงประชด 36% ของข้อความ**
ทั้งที่จริงมีแค่ 4.9% → FP 77 ข้อ, precision **0.115**, F1 ร่วงเหลือ **0.202**
บนข้อมูลจริง "จับให้ครบ" (recall 0.833) ไม่มีค่าถ้า 7 ใน 8 ที่จับเป็นของปลอม

**20.3 ผลที่สำคัญที่สุด: "อ่าน confidence ที่จ่ายไปแล้ว" คือสิ่งเดียวที่กู้ระบบให้ใช้ได้**

ปรับ threshold จาก logprob (ฟรี ไม่เพิ่ม call) บนชุดสุ่มจริง:
- flag rate ลดจาก 36% → **6%** (ใกล้ base rate จริง 4.9%)
- precision พุ่ง 0.115 → **0.429** (เกือบ 4 เท่า) · F1 พุ่ง 0.202 → **0.462**
- **นี่คือ +0.26 F1 ฟรี** — คันโยกที่ใหญ่ที่สุดในทั้งโปรเจกต์ ใหญ่กว่าการเพิ่ม agent ทุกแบบรวมกัน

GPT ยัง "มั่นใจเกิน" หนัก (prob สุดขั้ว 150/245 = 61%) แต่ threshold ก็ยังดึงกำไรมหาศาลได้
=> finding 1 ("ครึ่งหนึ่งของ 'ชัยชนะ multi-agent' คือ baseline ทิ้ง confidence ที่จ่ายไปแล้ว")
**พิสูจน์ครั้งสุดท้ายและหนักแน่นที่สุดตรงนี้**: บนข้อมูลจริง การอ่าน confidence ไม่ใช่ "อัปเกรดเล็กๆ"
แต่เป็น **เส้นแบ่งระหว่างระบบที่ใช้ได้กับใช้ไม่ได้** (F1 0.46 vs 0.20) — และมันฟรี

**บทเรียนรวบยอดของทั้งโปรเจกต์ (ตอกย้ำด้วยข้อมูลจริง):**
สิ่งที่คนคิดว่าต้องแก้ด้วยสถาปัตยกรรม (เพิ่ม agent, debate, verifier) แก้ไม่ได้เลยบนข้อมูลจริง —
ปัญหาคือ **rare class + over-flagging** ซึ่งแก้ได้ด้วยของฟรีที่มีอยู่แล้ว (logprob) ตัวเดียว
**absolute F1 ที่ดีที่สุดบนโลกจริง = 0.46 ไม่ใช่ 0.73** ที่เคยรายงาน → งานนี้ยากกว่าที่ทุก headline บอก

**ข้อควรระวัง:** ประชด 12 ข้อยังน้อย CI จึงกว้าง [0.182, 0.688] — ตัวเลข absolute อ่านเป็น "ระดับ" ได้
(0.4–0.5 ไม่ใช่ 0.7) แต่ยังฟันธง point estimate เป๊ะไม่ได้ · ชุดนี้ (`Gold/gold_random.csv`)
คือ test set ที่ distribution ตรงโลกจริงชุดแรกของโปรเจกต์ ใช้วัด absolute ของระบบใดๆ ได้เลย

## ข้อสรุปเชิงธีสิส (แก้ใหม่หลัง finding 7-12)

**เดิม:** "multi-agent (verifier อนุรักษ์ recall) ช่วยได้จริงแต่ marginal (+0.054, cost 1.80×)"
**ใหม่:** เมื่อเทียบกับ single agent ที่ **รีด logprob ออกมาใช้จนหมด** ข้อได้เปรียบนั้น **แยกจาก noise ไม่ออก**
บนงาน/ชุดข้อมูลนี้ (n ประชด = 30) **ไม่มีหลักฐานว่า multi-agent ใดๆ คุ้มค่าเงินที่จ่ายเพิ่ม**

**บทเรียนกลาง:** ก่อนถามว่า *"เพิ่ม agent ไหม"* ต้องถามก่อนว่า *"รีดของที่ agent เดียวให้มาแล้วหมดหรือยัง"*
การเทียบ multi-agent กับ baseline ที่ยังไม่ tune = การเทียบที่เอนเอียง เพราะปุ่มฟรี (threshold) ทำงานเดียวกัน
(ซื้อ precision แลก recall) ได้เกือบเท่ากันโดยไม่เพิ่ม call เลย

**ข้อควรระวัง (ไม่ใช่ว่า multi-agent ไร้ค่า):** error profile ต่างกันจริง
v2 รักษา recall 0.967 (พลาดประชด 1) ส่วน threshold ได้ 0.833 (พลาด 5) → งาน "คัดกรองห้ามพลาด" v2 ยังคุ้ม
สิ่งที่ finding นี้บอกคือ "ถ้าวัดที่ F1 ล้วน multi-agent ไม่ชนะ" ไม่ใช่ "multi-agent แพ้ทุกสถานการณ์"

**เดิม (เก็บไว้เป็นบันทึก):**

multi-agent (verifier อนุรักษ์ recall) **ช่วยได้จริง แต่ marginal**:
ได้ F1 +0.054 (precision +0.078, เสีย recall แค่ -0.033) แลกกับ **cost 1.80× / calls 1.44×**
"คุ้มไหม" = ขึ้นกับว่ายอมจ่ายเกือบเท่าตัวเพื่อ F1 เพิ่ม ~5 จุดไหม

คอขวดที่เหลือ = **precision** แม้ v2 ก็ยังปล่อย FP รอด 19 ข้อ (เพราะตั้งให้อนุรักษ์ recall)
มี precision/recall tension ที่ verifier ตัวเดียวแก้ไม่หมด -> ช่องว่างสำหรับงานต่อ

## ข้อควรระวังตอนเขียนรายงาน (สำคัญ)

- **อย่าใช้ accuracy** -- baseline acc 0.787 เกือบเท่าเดาว่า "ไม่ประชด" ทุกข้อ (0.764) เพราะข้อมูลเอียง
- ตัวเลข recall ของ GPT-4o อาจสูงเกินจริงจาก self-selection bias (ดู PROVENANCE.md);
  แต่ bias นี้กระทบทั้ง baseline และ multi-agent เท่ากัน -> การ "เทียบ" ยังยุติธรรม
- n ประชด = 30 น้อย ทุกข้อสรุปควรแนบ CI ไม่ใช่ point estimate เดี่ยว
- **ไม่มีที่ไหนใน repo ที่ pin `temperature` หรือ `seed` ของ OpenAI เลย** -> ทุก call รันที่ default
  temperature = 1.0 แปลว่า **ถ้ารันระบบเดิมซ้ำ ก็จะไม่ได้ prediction ชุดเดิม**
  ผลที่ตามมาต่อการรายงาน:
  - ตัวเลข F1 ของทุกระบบที่ใช้ GPT เป็น **ผลจากการรันครั้งเดียว** -> มี run-to-run noise ปนอยู่
  - `compare_systems.py` ทำ paired bootstrap **ข้ามข้อ** (item-level) ซึ่งจับ noise ตัวนี้ไม่ได้
    -> CI ที่รายงานไว้ **แคบกว่าความจริง** (ประเมินความมั่นใจสูงเกินไปเล็กน้อย)
  - ไม่ได้ทำให้การ *เทียบ* ระบบต่อระบบเสียไป เพราะทุกระบบที่ใช้ GPT โดน noise แบบเดียวกัน
    แต่ **ห้ามตีความผลต่างเล็กๆ (< ~0.03) ว่าเป็นของจริง** โดยไม่รันซ้ำหลายรอบ
  - งานต่อ: ตั้ง `temperature=0` (หรือ `seed=`) แล้วรันซ้ำ 3 รอบ เพื่อวัดว่า run-to-run SD เท่าไร
    ก่อนจะเคลมผลต่างระดับ 0.01-0.05 ระหว่างสถาปัตยกรรม
  (เจอตอนจะยืนยันว่า async debate ให้ผลเหมือน sequential ไหม -- **ยืนยันด้วยการรันซ้ำไม่ได้**
   เพราะ sequential เองก็ทำซ้ำตัวเองไม่ได้ ดู finding 17)
- เกณฑ์ "F1 > 0.793 (CI บนของ baseline)" ที่เคยพูดตอนแรก = เข้มเกินไป (เทียบแบบ unpaired)
  ที่ถูกคือ paired bootstrap ของผลต่าง ตามด้านบน

## ไฟล์ที่เกี่ยวข้อง

- `baseline.py` / `baseline_preds_gpt.csv` -- เอเจนต์เดี่ยว

- `wangchanberta.py` / `wangchanberta_preds.csv` -- โมเดลเล็ก (5-fold CV, ตัวเลขที่ใช้รายงาน)
- `wcb_oof_probs.py` / `wcb_oof_probs.csv` -- out-of-fold probability ของ WCB (ใช้หา threshold ในข้อ 6)
- `cascade.py` -- ระบบ ⑦ cascade (WCB คัดกรอง -> GPT verifier) · `--dry-run` ดูตารางข้อ 6 ได้โดยไม่เสียเงิน
- `train_final_wcb.py` / `wcb_model/` -- โมเดลตัวจริงสำหรับเว็บ (เห็น gold ครบ -> ห้ามใช้วัดผล)
- `compare_systems.py` -- paired bootstrap + McNemar (ครอบคลุมทั้ง 3 ระบบแล้ว)
- `gpt_threshold.py` / `gpt_screener_probs.csv` / `multiagent_preds_gpt_threshold.csv` -- ระบบ ⑧ (ข้อ 7)
  `--score` ยิง GPT เก็บ logprob (~$0.09) · `--sweep` วิเคราะห์ threshold (ฟรี)
- `frontier.py` / `frontier_probs_*.csv` / `frontier_meta_*.csv` -- cost-quality frontier 5 โมเดล (ข้อ 9)
  `--score` ยิงทุกโมเดล (~$0.20) · `--report` วิเคราะห์ F1/cost (ฟรี)
- `compare_fair.py` -- เทียบทุกระบบกับ baseline+threshold (ข้อ 8, ฟรี ใช้ pred เดิม)
- `analyze_baseline.py` -- แยกผลตามที่มาของข้อ (หลักฐาน self-selection bias)
- `app.py` -- เว็บทดลอง/เทียบ 3 ระบบสดๆ (ดู README_WEB.md)

**เพิ่มใน finding 15-18 (production / cost-aware):**
