# ผลการทดลอง: เอเจนต์เดี่ยว vs multi-agent (ประชดภาษาไทย)

วัดบน gold 127 ข้อ (ประชด 30 / ไม่ประชด 97) ทุกระบบใช้ GPT-4o + harness เดียวกัน
ที่มาของ gold และ self-selection bias: ดู `PROVENANCE.md`

## สรุปผล 3 ระบบ

| ระบบ | F1 | precision | recall | LLM calls | ค่าใช้จ่าย | latency p50 |
|---|---|---|---|---|---|---|
| **baseline** (เดี่ยว) | 0.690 | 0.526 | **1.000** | 127 | $0.094 | 751 ms |
| **baseline + threshold** (เดี่ยว, ปรับ threshold จาก logprob) ⭐ | 0.725 | 0.641 | 0.833 | 127 | **$0.094** | 751 ms |
| multi-agent **v1** (verifier ตื้น) | 0.714 | **0.769** | 0.667 | 180 | $0.157 | 721 ms |
| multi-agent **v2** (verifier อนุรักษ์ recall) | **0.744** | 0.604 | 0.967 | 183 | $0.169 | 967 ms |
| **debate** (อัยการ+ทนาย+ผู้พิพากษา) | 0.694 | 0.595 | 0.833 | 381 | $0.695 | 4557 ms |
| **cascade** (WCB คัดกรอง → GPT verifier) ✗ | 0.628 ± 0.006 | 0.500 | 0.844 | 119 | $0.124 | — |
| **WangchanBERTa** (5-fold CV × 3 seeds) | 0.620 ± 0.005 | 0.553 | 0.700 | **0** | **$0.00** | **26 ms** |

WangchanBERTa วัดด้วย 5-fold stratified CV (ทุกข้อถูกทำนายโดยโมเดลที่ไม่เคยเห็นข้อนั้น)
→ ได้ pred ครบ 127 ข้อ เทียบ paired กับ LLM ได้ตรงๆ · เทรนบน CPU 44 นาที · รันออฟไลน์ได้

## finding หลัก (ตอบธีสิส "multi-agent คุ้มไหม")

**1. baseline มี recall เต็ม (1.000) แต่ precision ต่ำ (0.526)**
GPT-4o จับประชดครบทุกข้อ แต่ตีความ "รีวิวสมดุล (ชมจริงติจริง)" เป็นประชดผิดๆ 27 ข้อ
-> ปัญหาไม่ใช่ "หาประชดไม่เจอ" แต่เป็น "เหวี่ยงแหกว้างเกิน"

**2. verifier เป็นด่านกรอง FP -- ปรับ prompt แล้วผลต่างกันมาก**
- v1 (ไม่ชัดก็พลิกทิ้ง): พลิก 27 ข้อ -- ถูก 17 ผิด 10 -> recall ร่วงเหลือ 0.667, F1 แทบไม่ขึ้น
- v2 (พลิกเฉพาะที่ชัดว่าไม่ใช่ประชด): พลิก 8 ข้อ -- เสีย recall แค่ 1 ข้อ -> F1 ดีสุด 0.744

**3. v2 ดีกว่า baseline แบบ "เกือบมีนัยสำคัญ" แต่แลกด้วยค่าใช้จ่ายเกือบเท่าตัว**
paired bootstrap (resample gold ชุดเดียวกัน 5000 รอบ):
- ผลต่าง F1 = +0.054, 95% CI [-0.002, +0.113]
- CI คร่อม 0 อยู่นิดเดียว (ขอบล่าง -0.002) -> โอกาสที่ v2 ไม่ดีกว่า = แค่ 3%
- McNemar: v2 ถูก-baseline ผิด 8 ข้อ / baseline ถูก-v2 ผิด 1 ข้อ (เอียงไปทาง v2 ชัด)
=> สรุปได้ว่า "มีแนวโน้มดีกว่าจริง (~97% confidence)" แต่ยังไม่ผ่านเกณฑ์ 95% แบบเป๊ะ
   ที่ n=30 ประชด กำลังสถิติไม่พอจะฟันธง

**4. agent มากขึ้น != ดีขึ้น -- debate แพ้ pipeline ที่เรียบง่ายกว่า ทั้งที่แพงกว่า 4 เท่า**
สถาปัตยกรรม debate (อัยการ + ทนาย -> ผู้พิพากษา, 3 calls/ข้อ, ตัดสินใหม่ได้ทั้งสองทาง):
- F1 **0.694** -- แทบไม่ต่างจาก baseline เอเจนต์เดี่ยว (+0.005, โอกาสไม่ดีกว่า = **46%** = โยนหัวก้อย)
- แพ้ pipeline v2 (0.744): ผลต่าง **-0.049**, โอกาสที่ debate ไม่ดีกว่า v2 = **83%**
- **แต่จ่ายแพงกว่า v2 ถึง 4.1× ($0.695 vs $0.169) และช้ากว่า 4.7× (4557 vs 967 ms)**

สาเหตุที่ debate แพ้ -- **มันคืนอิสระให้ตัดสินใหม่ทั้งสองทาง แล้วเสีย recall ฟรีๆ**:
recall ร่วงจาก 1.000 (baseline) เหลือ **0.833** -- ทนายแก้ต่างสำเร็จให้ประชดจริง 5 ข้อ (FN=5)
ขณะที่ verifier ของ v2 **โครงสร้างบังคับ** ให้พลิกได้ทางเดียว (1->0) และตั้งกฎ "ไม่ชัดให้คงไว้"
=> v2 จึงรักษา recall 1.000 ที่ detector ได้มาฟรีไว้เกือบครบ (0.967) แล้วค่อยๆ ซื้อ precision

**บทเรียนเชิงสถาปัตยกรรม (สำคัญกว่าตัวเลข):**
**"ข้อจำกัดของ agent" มีค่ามากกว่า "จำนวน agent"**
verifier ที่ทำได้อย่างเดียวคือปัดตก (จำกัดมาก) เอาชนะคณะ 3 agent ที่ตัดสินอิสระได้ทุกทาง
เพราะมันหวงของดีที่มีอยู่แล้ว (recall เต็ม) แทนที่จะเอาไปเสี่ยงใหม่ทุกข้อ

**5. โมเดลเล็กฟรี "ยังตัดทิ้งไม่ได้" -- และนี่คือผลที่ไม่คาดคิดที่สุด**
WangchanBERTa (F1 0.620) ดูแพ้ baseline (0.690) ที่ผิวเผิน แต่พอทดสอบแบบ paired:
- ผลต่าง F1 = -0.072, 95% CI **[-0.206, +0.053]** -> **คร่อม 0**
- McNemar: WCB ถูก-baseline ผิด **15** ข้อ | baseline ถูก-WCB ผิด **14** ข้อ -> **แทบเสมอ**
=> ที่ n=30 **ยังฟันธงไม่ได้ว่า GPT-4o ชนะโมเดลฟรีที่เทรนด้วยข้อมูลแค่ 127 ข้อจริง**

แต่ **อย่าตีความเกินจริง**: point estimate ยังบอกว่า WCB แย่กว่า (โอกาสที่มันแย่กว่า = 87%)
"สรุปไม่ได้ว่าแย่กว่า" != "ดีเท่ากัน" -- มันแค่แปลว่า gold เล็กเกินกว่าจะตัดสิน
และ **error profile ต่างกันชัด**: WCB ปล่อยประชดหลุด 9 ข้อ (recall 0.700) ขณะที่ LLM ไม่หลุดเลย
-> ถ้างานคือ "คัดกรองห้ามพลาด" LLM ยังชนะขาด แม้ F1 จะไม่ต่างกันอย่างมีนัยสำคัญ

ข้อดีที่วัดได้ของ WCB: **$0.00 · 0 API calls · 26 ms (เร็วกว่า LLM 29 เท่า) · รันออฟไลน์ได้**

**6. เอา WangchanBERTa (ฟรี) มาเป็นด่านคัดกรองแทน GPT — "ไม่ได้ผล" และเหตุผลที่ล้มเหลวคือ finding เอง**

ไอเดีย (ตรงตามบทเรียนข้อ 4 เป๊ะ): สถาปัตยกรรมที่ชนะคือ *ด่านคัดกรอง recall สูง → verifier ที่ปัดตกได้อย่างเดียว*
แต่ v2 ใช้ **GPT เป็นด่านคัดกรอง = จ่ายเงินทุกข้อ (127 calls)** ซึ่งเป็น "พื้นราคา" ที่ตัดไม่ได้เลย
ถ้าเปลี่ยนด่านแรกเป็น WangchanBERTa (ฟรี · ออฟไลน์ · 26 ms) พื้นราคานั้นควรหายไปทั้งชั้น

**F1 0.620 ของ WCB ไม่ใช่เหตุผลที่มันเป็นด่านคัดกรองไม่ได้** — ด่านคัดกรองไม่ต้องแม่น ต้องแค่ "ไม่ปล่อยประชดหลุด"
(recall สูง) ส่วนที่เหวี่ยงเกิน verifier ตัดทิ้งให้ทีหลัง → จึงต้องวัดที่ **recall ตอนลด threshold** ไม่ใช่ F1 ที่ 0.5

วิธีวัด: ผลิต out-of-fold probability (`wcb_oof_probs.py`, 5-fold × 3 seeds — โปรโตคอลเดียวกับข้อ 5)
แล้วเลือก threshold แบบ **leave-fold-out** (ข้อใน fold k ใช้ threshold ที่เลือกจากอีก 4 folds เท่านั้น)
→ ไม่มีข้อไหนมีส่วนกำหนด threshold ที่ตัดสินตัวมันเอง (กัน threshold leak)

| target recall | flag (จาก 127) | recall จริง | precision ด่านแรก | GPT calls | ค่าใช้จ่าย | vs v2 |
|---|---|---|---|---|---|---|
| 0.85 | 81 | 0.822 | 0.31 | 81 | $0.085 | 0.50× |
| 0.90 | 92 | 0.889 | 0.29 | 92 | $0.096 | 0.57× |
| 0.95 | 105 | 0.933 | 0.27 | 105 | $0.110 | 0.65× |
| 1.00 | **119** | 0.967 | **0.24** | 119 | $0.124 | 0.74× |

**ตัวเลขที่ฆ่าไอเดียนี้: เพื่อจับประชดให้ได้ 29/30 ข้อ ด่านคัดกรองต้อง flag 119 จาก 127 ข้อ = 94% ของทั้งชุด**

- ส่งทุกข้อให้ verifier ตรงๆ โดยไม่ต้องคัดกรองเลย = 127 calls, recall 1.000
- ให้ WCB คัดกรองก่อน = 119 calls, recall 0.967
- → **ด่านคัดกรองฟรีนี้ประหยัดได้ 8 calls แล้วทำประชดจริงหลุดไป 1 ข้อเป็นค่าตอบแทน**
  มันไม่ได้ "กรอง" อะไรเลย มันแค่เพิ่มขั้นตอน

เทียบกับ GPT ที่ทำงานเดียวกัน: recall 1.000 โดย flag แค่ **57 ข้อ** (precision 0.526)
WCB ต้อง flag มากกว่า **2 เท่า** เพื่อให้ได้ recall **ต่ำกว่า** ที่ precision ครึ่งเดียว (0.24)
และ verifier ของ v2 ถูกออกแบบให้อนุรักษ์นิยม ("ไม่ชัด → คงไว้") — ใน v2 มันปัดตกแค่ 8 จาก 56 ข้อที่เห็น (14%)
→ โยน FP ให้มัน 89 ข้อแทนที่จะเป็น 27 ข้อ **มันกู้ precision ไม่ไหวแน่นอน**

**บทเรียน: "ฟรี" ไม่ได้แปลว่า "เอามาวางหน้า pipeline แล้วประหยัด"**
ด่านคัดกรองที่ใช้ได้ต้อง **จัดอันดับ (rank) เป็น** — ต้องแยกได้ว่าข้อไหน "ชัดว่าไม่ประชด" เพื่อจะตัดทิ้งฟรี
WCB ที่เทรนด้วย 127 ข้อ **ทำไม่ได้**: คะแนนของมันแทบไม่มีข้อมูลตรงหางที่ต้องใช้ตัดสิน
→ ยืนยันข้อ 5 อีกทาง: โมเดลเล็กฟรี "ยังตัดทิ้งไม่ได้" — ไม่ใช่แค่ในฐานะระบบเดี่ยว แต่ในฐานะ **ชิ้นส่วน** ของระบบด้วย

**ยืนยันด้วยการรันจริง (ไม่ใช่แค่ประมาณการ):** รัน verifier จริงบนข้อที่ WCB flag (target recall 1.00)
→ **F1 0.628 ± 0.006** (3 seeds) · 119 calls · $0.124
- **แย่กว่า baseline เอเจนต์เดี่ยว (0.690) และ *แพงกว่า* ด้วย ($0.124 vs $0.094)** — แพ้ทุกแกน
- paired bootstrap vs baseline: ผลต่าง **-0.062**, 95% CI [-0.152, +0.025] · McNemar: base ถูก-cascade ผิด 14 / cascade ถูก-base ผิด 9
- (ตอนดูแค่ dry-run เคยเดาว่า F1 จะร่วงถึง ~0.42 — เดาแรงเกินไป verifier ปัดตกได้เยอะกว่าที่คิด
  แต่ **ข้อสรุปไม่เปลี่ยน**: จ่ายแพงกว่า baseline เพื่อได้ F1 ต่ำกว่า)

*(หมายเหตุการทำซ้ำ: รอบนี้ WCB @0.5 ได้ F1 0.606 / 0.576 / 0.588 → mean **0.590** (SD 0.012)
ต่ำกว่าที่รายงานไว้ข้อ 5 (0.620 ± 0.005) และแกว่งกว่า — น่าจะมาจาก torch/transformers คนละเวอร์ชัน
แต่แปลว่าตัวเลข WCB ในตารางอาจ "อ่อน" กว่าที่เห็น และ SD ±0.005 อาจต่ำเกินจริง)*

**7. เอเจนต์เดี่ยว + ปรับ threshold ≈ multi-agent v2 — ที่ราคาครึ่งเดียว (finding ที่สั่นธีสิสที่สุด)**

ทุกระบบข้างบนตอบ 0/1 ตายตัว "ที่จุดทำงานจุดเดียว" — แต่ GPT ไม่ได้ตอบ 0/1 จริงๆ มันตอบ *token* ที่มี logprob
ดึง P("1") ออกมาจาก logprob ของ token label ได้เลย (`gpt_threshold.py`) โดย **ไม่แตะ prompt และไม่เพิ่ม call แม้แต่ครั้งเดียว**
→ คะแนนนี้ได้มา "ฟรี" จาก call ที่จ่ายไปแล้วใน baseline

GPT **จัดอันดับเป็น** (ต่างจาก WCB ในข้อ 6 อย่างสิ้นเชิง) — เลื่อน threshold แล้วซื้อ precision ได้จริง:

| tau | precision | recall | F1 |
|---|---|---|---|
| 0.50 (argmax = baseline เดิม) | 0.526 | **1.000** | 0.690 |
| 0.75 | 0.558 | 0.967 | 0.707 |
| 0.95 | 0.596 | 0.933 | 0.727 |
| 0.99 | 0.667 | 0.800 | 0.727 |

เลือก threshold แบบ **leave-fold-out** (tau ≈ 0.984 ทุก fold) → **F1 0.725 · prec 0.641 · recall 0.833 · 127 calls · $0.094**

**เทียบกับ v2 (2 agents) แบบ paired — นี่คือคำถามเงินล้วนๆ ว่า "agent ที่ 2 คุ้มไหม":**
- v2 0.744 vs threshold 0.725 → ผลต่าง **+0.019**, 95% CI **[-0.076, +0.121]** → **คร่อม 0 กว้างมาก**
- โอกาสที่ v2 **ไม่ได้** ดีกว่า = **34%**
- McNemar: v2 ถูก-threshold ผิด **7** ข้อ | threshold ถูก-v2 ผิด **8** ข้อ → **เสมอกันสนิท**
- **แต่ v2 แพงกว่า 1.80× (+$0.075, +56 calls)**

=> **agent ตัวที่ 2 ซื้อ F1 ที่แยกจาก noise ไม่ออก ด้วยราคาเกือบเท่าตัว**
   สิ่งที่ v2 ทำได้จริง (ซื้อ precision โดยเสีย recall ให้น้อยที่สุด) **ปุ่ม threshold ตัวเดียวก็ทำได้เกือบเท่ากัน — ฟรี**

**ข้อควรระวัง (อย่ารีบเปลี่ยนไปใช้ threshold):** โปรไฟล์ error ต่างกันชัด
- v2: recall **0.967** (พลาดประชด 1 ข้อ) — ยังรักษาของดีจาก detector ไว้
- threshold: recall **0.833** (พลาดประชด 5 ข้อ) — มันซื้อ precision ด้วยการ *ยอมทิ้ง recall*
→ ถ้างานคือ "คัดกรองห้ามพลาด" **v2 ยังชนะ** และ 0.075 ดอลลาร์นั้นคุ้ม
→ ถ้าวัดกันที่ F1 ล้วน **ปุ่มฟรีชนะขาดในเชิงราคา**

**บทเรียน (ต่อจากข้อ 4):** ก่อนจะถามว่า *"เพิ่ม agent ไหม"* ต้องถามก่อนว่า
***"รีดของที่ agent ตัวเดียวให้มาอยู่แล้ว ออกมาหมดหรือยัง"*** — ที่นี่คำตอบคือ **ยัง**
baseline ทิ้ง logprob (ข้อมูลที่จ่ายเงินซื้อมาแล้ว) ลงถังขยะทุกข้อ แล้วเราไปซื้อ agent ตัวที่ 2 มาทดแทน
**การเทียบ multi-agent กับ baseline ที่ยังไม่ถูก tune ให้สุด = การเทียบที่ให้เครดิต multi-agent เกินจริง**

**8. การเทียบที่ยุติธรรม: เทียบทุกระบบกับ baseline ที่ tune แล้ว → ไม่มีระบบไหนชนะแบบมีนัยสำคัญเลย**

finding 1-7 เทียบทุกอย่างกับ baseline @argmax (0.690) ซึ่ง "ทิ้ง logprob" ที่จ่ายเงินซื้อมาแล้ว
= ให้ multi-agent สู้กับคู่ต่อสู้ที่ถูกมัดมือ คู่เทียบที่ถูกคือ **baseline+threshold (0.725, ราคาเท่าเดิม)**
รันใหม่ทั้งหมด (`compare_fair.py`, paired bootstrap, ไม่ยิง API ใช้ pred เดิม):

| ระบบ | F1 | ต่างจาก baseline+threshold | 95% CI | P(ไม่ดีกว่า) | McNemar | ราคา |
|---|---|---|---|---|---|---|
| conservative (v2) | 0.744 | +0.019 | [-0.073, +0.113] | 36% | 7-8 | $0.169 |
| v1aggressive | 0.714 | -0.010 | [-0.159, +0.135] | 57% | 11-8 | $0.157 |
| hybrid | 0.700 | -0.025 | [-0.117, +0.071] | 70% | 6-11 | $0.407 |
| debate | 0.694 | -0.030 | [-0.158, +0.096] | 69% | 9-12 | $0.695 |
| cascade | 0.628 | -0.097 | [-0.203, +0.012] | 96% | 7-20 | $0.124 |
| wangchanberta | 0.618 | -0.107 | [-0.235, +0.018] | 95% | 7-14 | $0.00 |

**ทุกระบบ CI คร่อม 0 — ไม่มีระบบใดแยกออกจาก baseline+threshold ได้อย่างมีนัยสำคัญ**
ระบบ multi-agent ที่ดีที่สุด (v2) ยังมีโอกาสถึง 36% ที่ *ไม่ได้* ดีกว่าปุ่ม threshold ตัวเดียว ที่ราคา 1.80×
debate/hybrid ที่แพงกว่า 4-7× ก็อยู่ฝั่ง *ต่ำกว่า* ปุ่มฟรีด้วยซ้ำ

=> **ข้อได้เปรียบของ multi-agent v2 ที่เคยรายงานว่า "+0.054 vs baseline" หดเหลือ +0.019 vs baseline ที่ tune แล้ว
และหายไปใน 노이즈** ครึ่งหนึ่งของ +0.054 เดิม คือ "baseline ถูก handicap" ไม่ใช่ "multi-agent เก่ง"

**9. cost-quality frontier: เปลี่ยน "โมเดล" ขยับ cost ได้ 25× โดย F1 แทบไม่ขยับ — และโมเดลถูกได้ F1 สูงสุด**

finding 1-8 อยู่บน gpt-4o โมเดลเดียว โปรเจกต์ชื่อ "Cost-Aware" จึงควรวัดแกน *โมเดล* ด้วย ไม่ใช่แค่แกนสถาปัตยกรรม
เอาสถาปัตยกรรมที่ดีสุด+ถูกสุด (เอเจนต์เดี่ยว + threshold, finding 7) ไปรันบน 5 โมเดลที่ราคาต่างกัน 25×
(`frontier.py`, threshold เลือกแบบ leave-fold-out เหมือนเดิม):

| โมเดล | F1 | prec | recall | $/รัน (127 ข้อ) | $/1M in |
|---|---|---|---|---|---|
| gpt-4.1-nano | 0.706 | 0.632 | 0.800 | $0.0038 | $0.10 |
| gpt-4o-mini | 0.676 | 0.568 | 0.833 | $0.0056 | $0.15 |
| **gpt-4.1-mini** ⭐ | **0.727** | 0.667 | 0.800 | $0.0150 | $0.40 |
| gpt-4.1 | 0.716 | 0.649 | 0.800 | $0.0752 | $2.00 |
| gpt-4o | 0.725 | 0.641 | 0.833 | $0.0940 | $2.50 |

- **F1 อยู่ในช่วง 0.68-0.73 ตลอดทั้งแถบราคา 25×** — ช่วง 0.05 นี้เล็กกว่า CI ที่ n=30 (±0.10) → แทบทั้งหมดคือ noise
- **โมเดลที่ได้ F1 สูงสุดคือ gpt-4.1-mini ($0.015) ไม่ใช่ gpt-4o ($0.094)** — โมเดลที่ทั้งโปรเจกต์สร้างอยู่บนนั้น ไม่ใช่จุดที่ดีที่สุดด้วยซ้ำ
- gpt-4o-mini เป็นตัวหลุด (F1 ต่ำสุด 0.676) → F1 ไม่ได้ไล่ตามราคาแบบเส้นตรง เลือกโมเดลถูกต้องลองจริง ไม่ใช่เดาจากราคา

**หมัดเด็ด — เอเจนต์เดี่ยวโมเดลถูก vs multi-agent เรือธง (paired, n=127):**

| | F1 | ราคา | agents / calls |
|---|---|---|---|
| gpt-4.1-mini + threshold | 0.727 | **$0.015** | 1 agent / 127 |
| gpt-4o v2 (multi-agent เรือธง) | 0.744 | $0.169 | 2 agents / 183 |

ผลต่าง +0.016, 95% CI **[-0.094, +0.135]** (คร่อม 0) · P(v2 ไม่ดีกว่า) = **39%** · **v2 แพงกว่า 11×**

=> ระบบ multi-agent ที่ดีที่สุดของโปรเจกต์ **แยกจากเอเจนต์เดี่ยวบนโมเดลถูกไม่ออก แต่แพงกว่า 11 เท่า**
   "cost-aware" ตัวจริงของงานนี้ไม่ใช่ "เลือกสถาปัตยกรรม" แต่คือ **"เลือกโมเดลถูก + อ่าน logprob ของมัน"**

*(ราคาเป็นค่าประมาณใน `frontier.py`/PRICE ต้องตรวจกับราคาปัจจุบัน; token counts วัดจริงจาก API เป๊ะ 34,535 in / 762 out เท่ากันทุกโมเดล)*

## ข้อสรุปเชิงธีสิส (แก้ใหม่หลัง finding 7-9)

**เดิม:** "multi-agent (verifier อนุรักษ์ recall) ช่วยได้จริงแต่ marginal (+0.054, cost 1.80×)"
**ใหม่:** เมื่อเทียบกับ single agent ที่ **รีด logprob ออกมาใช้จนหมด** ข้อได้เปรียบนั้น **แยกจาก noise ไม่ออก**
บนงาน/ชุดข้อมูลนี้ (n ประชด = 30) **ไม่มีหลักฐานว่า multi-agent ใดๆ คุ้มค่าเงินที่จ่ายเพิ่ม**

**บทเรียนกลาง:** ก่อนถามว่า *"เพิ่ม agent ไหม"* ต้องถามก่อนว่า *"รีดของที่ agent เดียวให้มาแล้วหมดหรือยัง"*
การเทียบ multi-agent กับ baseline ที่ยังไม่ tune = การเทียบที่เอนเอียง เพราะปุ่มฟรี (threshold) ทำงานเดียวกัน
(ซื้อ precision แลก recall) ได้เกือบเท่ากันโดยไม่เพิ่ม call เลย

**ข้อควรระวัง (ไม่ใช่ว่า multi-agent ไร้ค่า):** error profile ต่างกันจริง
v2 รักษา recall 0.967 (พลาดประชด 1) ส่วน threshold ได้ 0.833 (พลาด 5) → งาน "คัดกรองห้ามพลาด" v2 ยังคุ้ม
สิ่งที่ finding นี้บอกคือ "ถ้าวัดที่ F1 ล้วน multi-agent ไม่ชนะ" ไม่ใช่ "multi-agent แพ้ทุกสถานการณ์"

**เดิม (เก็บไว้เป็นบันทึก):**

multi-agent (verifier อนุรักษ์ recall) **ช่วยได้จริง แต่ marginal**:
ได้ F1 +0.054 (precision +0.078, เสีย recall แค่ -0.033) แลกกับ **cost 1.80× / calls 1.44×**
"คุ้มไหม" = ขึ้นกับว่ายอมจ่ายเกือบเท่าตัวเพื่อ F1 เพิ่ม ~5 จุดไหม

คอขวดที่เหลือ = **precision** แม้ v2 ก็ยังปล่อย FP รอด 19 ข้อ (เพราะตั้งให้อนุรักษ์ recall)
มี precision/recall tension ที่ verifier ตัวเดียวแก้ไม่หมด -> ช่องว่างสำหรับงานต่อ

## ข้อควรระวังตอนเขียนรายงาน (สำคัญ)

- **อย่าใช้ accuracy** -- baseline acc 0.787 เกือบเท่าเดาว่า "ไม่ประชด" ทุกข้อ (0.764) เพราะข้อมูลเอียง
- ตัวเลข recall ของ GPT-4o อาจสูงเกินจริงจาก self-selection bias (ดู PROVENANCE.md);
  แต่ bias นี้กระทบทั้ง baseline และ multi-agent เท่ากัน -> การ "เทียบ" ยังยุติธรรม
- n ประชด = 30 น้อย ทุกข้อสรุปควรแนบ CI ไม่ใช่ point estimate เดี่ยว
- เกณฑ์ "F1 > 0.793 (CI บนของ baseline)" ที่เคยพูดตอนแรก = เข้มเกินไป (เทียบแบบ unpaired)
  ที่ถูกคือ paired bootstrap ของผลต่าง ตามด้านบน

## ไฟล์ที่เกี่ยวข้อง

- `baseline.py` / `baseline_preds_gpt.csv` -- เอเจนต์เดี่ยว
- `multiagent.py` / `multiagent_preds_gpt_conservative.csv` (v2) / `..._v1aggressive.csv` (v1)
- `wangchanberta.py` / `wangchanberta_preds.csv` -- โมเดลเล็ก (5-fold CV, ตัวเลขที่ใช้รายงาน)
- `wcb_oof_probs.py` / `wcb_oof_probs.csv` -- out-of-fold probability ของ WCB (ใช้หา threshold ในข้อ 6)
- `cascade.py` -- ระบบ ⑦ cascade (WCB คัดกรอง -> GPT verifier) · `--dry-run` ดูตารางข้อ 6 ได้โดยไม่เสียเงิน
- `train_final_wcb.py` / `wcb_model/` -- โมเดลตัวจริงสำหรับเว็บ (เห็น gold ครบ -> ห้ามใช้วัดผล)
- `compare_systems.py` -- paired bootstrap + McNemar (ครอบคลุมทั้ง 3 ระบบแล้ว)
- `gpt_threshold.py` / `gpt_screener_probs.csv` / `multiagent_preds_gpt_threshold.csv` -- ระบบ ⑧ (ข้อ 7)
  `--score` ยิง GPT เก็บ logprob (~$0.09) · `--sweep` วิเคราะห์ threshold (ฟรี)
- `frontier.py` / `frontier_probs_*.csv` / `frontier_meta_*.csv` -- cost-quality frontier 5 โมเดล (ข้อ 9)
  `--score` ยิงทุกโมเดล (~$0.20) · `--report` วิเคราะห์ F1/cost (ฟรี)
- `compare_fair.py` -- เทียบทุกระบบกับ baseline+threshold (ข้อ 8, ฟรี ใช้ pred เดิม)
- `analyze_baseline.py` -- แยกผลตามที่มาของข้อ (หลักฐาน self-selection bias)
- `app.py` -- เว็บทดลอง/เทียบ 3 ระบบสดๆ (ดู README_WEB.md)
